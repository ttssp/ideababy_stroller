### TestStaleSaveTagRebinding(R1 H2 + R2 B-R2-3 fix)
**场景**:operator 打开 friction-log 编辑器中(stale snapshot)→ hook 后台 append 新 entry(file 已变化)→ operator 保存覆盖(stale write)→ 新 entry 丢 + 旧 tag 绑错位置
1. 准备 friction-log 含 entry E1 + tag T1;`update(enabled=True)`
2. 模拟 operator 读出 friction-log 内容(snapshot)
3. 触发 hook append 新 entry E2(file 现含 E1 / T1 / E2)
4. operator 修改 snapshot(在 T1 后追加 T1' tag)+ 写回 file → E2 丢失,T1' 物理绑错
5. **断言 1**:此场景下不跑 `friction --review-mode` → file 内容确实丢 E2(证明 race 真实存在)
6. **断言 2**:跑 `friction --review-mode`(R2 B-R2-3:`set_review_mode(True)` 设 review_mode=true,**enabled 仍为 true**)→ hook gate 短路(`enabled AND NOT review_mode`)→ hook 不再 fire,operator snapshot 与 file 一致 → 写回安全
7. **断言 3**:operator 编辑完 → `friction --review-mode-off` 退出 review_mode,后续 hook 恢复正常(**不需要 `--on`,因为 enabled 从未变过**)
8. **断言 4(R2 B-R2-3)**:跑完整 review session 后 `state.json` 应见 `enabled: true` AND `review_mode: false`(终态正确)— 不再像 R1 方案那样最终 enabled 被改回 true(因为从来没被设为 false)

### TestO3PassesDuringReviewMode(R2 B-R2-3 / D20 fix · 新增)
**场景**:O3 与 review_mode 正交;review session 期间跑 day-14 trust_summary `--check-enabled` 仍 PASS
1. `update(enabled=True)`(operator 已启动 dogfood)
2. `set_review_mode(True)`(operator 进 review session)
3. `subprocess.run(["python", "-m", "friction_tap.trust_summary", "--check-enabled"])` → returncode == 0(**O3 PASS**)
4. stdout 含 `enabled: true`(**显式确认 enabled 仍 true,不受 review_mode 影响**)
5. **反向断言**:`update(enabled=False)` + `set_review_mode(False)` → `--check-enabled` returncode == 1(确认 O3 binary 仍工作)
6. **关键反向断言(对比 R1 方案的 false fail)**:`update(enabled=True)` + `set_review_mode(True)` 与 `update(enabled=False)` 是**两个不同 state**;前者 `--check-enabled` exit 0(R2 B-R2-3 fix),后者 exit 1 — R1 review-mode-as-off-alias 方案会让两者都 exit 1(false fail);R2 修订让 review session 不破 O3

### TestOnClearsReviewMode(R3/R4 H-R3-1 / D21 fix · 新增)
**场景**:`friction --on` 是"完整恢复"语义 — 同时清 review_mode,防 enabled=true / review_mode=true 卡死(hook 仍静默暂停 + O3 PASS 的 false adoption 状态)
1. **核心场景 a · operator 习惯敲 `--on` 退出 review**:
   - `update(enabled=False)` + `set_review_mode(True)`(模拟 operator 之前关了 hook 又进 review session)
   - `subprocess.run(["python", "-m", "friction_tap.cli", "--on"])` → returncode == 0
   - 断言:`state.load()` 显示 `enabled == True` AND `review_mode == False`(**双字段都恢复**)
   - 断言:`state.load().review_mode_set_at is None`(**R4-cleanup L-R4-3:timestamp 被显式清空** — 防止 stale set_at 触发未来 trust_summary 误算 24h WARNING)
   - 断言:stdout 含 `review_mode cleared`(operator 显式看到 review_mode 也被清)
2. **核心场景 b · review_mode 已 false 时无重复噪音**:
   - `update(enabled=False)` + `set_review_mode(False)`(干净中间态)
   - `subprocess.run(["python", "-m", "friction_tap.cli", "--on"])` → returncode == 0
   - 断言:stdout **不含** `review_mode cleared`(无重复;只 print `enabled · log: <path>`)
3. **反向断言 · `--off` 不清 review_mode**:
   - `update(enabled=True)` + `set_review_mode(True)`
   - `subprocess.run(["python", "-m", "friction_tap.cli", "--off"])` → returncode == 0
   - 断言:`state.load()` 显示 `enabled == False` AND `review_mode == True`(`--off` 仅关 enabled,**不动** review_mode — 与 `--on` 完整恢复非对称 · D21 注释)
4. **反向断言 · `--review-mode-off` 不动 enabled**:
   - `update(enabled=False)` + `set_review_mode(True)`
   - `subprocess.run(["python", "-m", "friction_tap.cli", "--review-mode-off"])` → returncode == 0
   - 断言:`state.load()` 显示 `enabled == False`(从未变)AND `review_mode == False`(已清)— 部分恢复
   - 断言:`state.load().review_mode_set_at is None`(**R4-cleanup L-R4-3:`--review-mode-off` 也走 set_review_mode(False) helper,timestamp 一并清空** — 与 architecture §5.4 / set_review_mode invariant 一致)

### TestTrustSummaryShowsReviewMode(R3/R4 H-R3-2 fix · 新增)
**场景**:trust_summary 输出需显示 `state.review_mode` 当前值,防 day-14 review 看到 O3 PASS 但其实 review_mode 长期 stuck true 致 hook 静默暂停
1. **基础渲染含 review_mode 状态**:
   - `update(enabled=True)` + `set_review_mode(True)`
   - `subprocess.run(["python", "-m", "friction_tap.trust_summary"])` → returncode == 0
   - 断言:trust-w2.md 含 `## Hook Operational Status` 段(或等价 `## Review Mode Status` 命名 · architecture §5.5)
   - 断言:该段显示 `state.review_mode = true`(明文)
   - 断言:O3 verdict 段独立 — `o3_verdict: pass`(D20 正交 · review_mode=true 不破 O3)
2. **review_mode = false 也显示**(避免段缺失致歧义):
   - `set_review_mode(False)`
   - 重跑 trust_summary → trust-w2-2.md 含 `state.review_mode = false`
3. **stuck-true 警告**(若实装 ≥ 24h 触发 WARNING 文案):
   - 实装 hint:在 `state.set_review_mode(True)` 时附记 `review_mode_set_at` ISO 时间戳到 state.json(architecture §5.4 加 field;backward-compat:缺失视为 None,跳过 24h 计算);trust_summary 读该 field,若现在 - set_at > 24h → 在 `## Hook Operational Status` 段加 `WARNING: review_mode has been on for >24h; hook is paused. Run \`friction --review-mode-off\` if review session is over.`
   - 测试用 monkeypatch 把 review_mode_set_at 设回 25h 前 → 跑 trust_summary → 断言 trust-w2.md 含 `WARNING` 与 `>24h`

### TestSchemaDriftSmokeTest(R1 H3 fix · 新增)
**场景**:install.sh 之后跑一次 real-payload smoke test 验证 hook 写入正确,且 schema_error_count 在故意 drift payload 下累加
1. **正向**:`echo '{"tool_name":"Bash","exit_code":1,"stderr":"Error: build failed: missing config"}' | python -m friction_tap.hook` → friction-log 末尾多一行,exit 0,state.json schema_error_count 不变
2. **反向(drift)**:`echo '{"toolName":"Bash"}' | python -m friction_tap.hook`(用旧字段名 toolName 模拟 schema drift) → exit 0(TECH-4 不破),state.json schema_error_count 从 N 变 N+1,last_run.log 末尾多 traceback
3. **空 stdin**:`echo '' | python -m friction_tap.hook` → exit 0,schema_error_count 不增(空 stdin 是合法 noop)
4. **trust-summary 显示 schema_error**:跑 reverse case 后跑 trust-summary → trust-w2.md `## Schema health` 段显示 `schema_error_count: 1`

## Known gotchas

- `subprocess.run` 启动 Python 解释器有 overhead(~80ms × 多次)→ 用 `unittest.skipUnless` + `os.environ.get("INTEGRATION") == "1"` 把慢测分组,或合并多步骤减少 subprocess 数
- 并发测试在 NFS / iCloud Drive 不保证(TECH-5)— 必须用 macOS local APFS tempdir(`tempfile.mkdtemp()` 默认 `/tmp` 是 APFS)
- monkeypatch `paths.state_path` 和 `paths.friction_log_path` 必须在 subprocess 外的环境变量传递(stdlib 没有 `subprocess.patch`)→ 用 `env={"PYTHONPATH": ..., "FRICTION_TAP_STATE": ..., ...}`,并在 `paths.py` 加可选 env override(本 task 可微调 paths.py 加 env hook,与 T001 file_domain 重叠时由 worker 协调)
- worktree 测试在 CI(无 git installed)会 fail — README 注明 macOS dev box only
- 不要在 fixture log 写未来时间戳 → trust_summary 的 `count_since(timedelta(days=14))` 会过滤未来,导致测试不稳
