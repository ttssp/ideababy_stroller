## Implementation plan

### README.md 章节(顺序敏感)

1. **Title + 一句话定位**:`friction-tap v0.1 — Claude Code session 的 friction 协作摩擦 archive(single-operator dogfood substrate)`
2. **What this is**:三句话 — agent-emit + operator-cli 双声部 archive,markdown + grep 即审阅;day-14 trust 报告 + 1 题 self-interview hedge
3. **What this is NOT**(COMM-1 + non-goals 1.6/1.7/1.8):
   - 不与 enterprise observability 工具竞争(Datadog / Langfuse / etc.)
   - 不分析 / 不打分 / 不开 issue / 不下结论(agent 是 witness · D11)
   - 不上传 / 不外发 / 不团队共享(default private · C10)
4. **Install · 5 步起点仪式**(TECH-3 mitigation):
   ```bash
   git clone <repo> && cd <repo>
   bash projects/007-007a-pA/scripts/install.sh   # T033 实现 — 把 hook + bin/friction 链好
   bash projects/007-007a-pA/scripts/install_launchd.sh  # day-14 trigger
   friction --on   # 启动 dogfood mode(D15 / ADR-5;不跑这条 hook 全 noop)
   echo "docs/dogfood/v4-friction-log.md" >> .gitignore  # SEC-1 推荐
   echo "docs/dogfood/v4-trust-w2.md" >> .gitignore
   ```
5. **Daily commands**:
   - `friction "<msg>"` 主观体感(IN-6)
   - `friction --threshold high` 收紧(TECH-2 mitigation · **R1 B1 修订后真过滤 M/L event**)
   - **`friction --on`(R3/R4 H-R3-1 / D21 修订 · "完整恢复"语义)**:`set_enabled(true)` AND `set_review_mode(false)` — 同时打开 hook 与清 review_mode,确保不会出现 enabled=true / review_mode=true 卡死(hook 仍静默暂停 + O3 PASS 的 false adoption 状态);若 review_mode 之前为 true → stdout 含 `review_mode cleared`(显式确认);若已 false → stdout 仅 `enabled`(无重复)。**operator 习惯 `--on` 退出 review session 是安全的**
   - `friction --off` 一行止血(让 enabled=false · adoption signal 关 · **不动** review_mode — 与 `--on` 完整恢复非对称)
   - **`friction --review-mode` / `--review-mode-on`(R1 H2 + R2 B-R2-3 / D17 / D20 / C14)**:**设 `state.review_mode=true`(与 enabled 正交,不动 adoption signal · D20)**;**审阅 friction-log 期间必跑此命令**抑制 hook,避免 stale-save tag 误绑;**与 `--off` 不同**:`--off` 让 adoption signal=false;`--review-mode` 仅暂停 hook 不影响 adoption — review session 期间 day-14 O3 仍 PASS
   - **`friction --review-mode-off`(R2 B-R2-3)**:退出 review_mode 仅(只清 review_mode,不动 enabled · 部分恢复);适用于 review session 结束但 dogfood mode 一直 enabled 的场景
   - **何时用 `--on` vs `--review-mode-off`(R3/R4 H-R3-1)**:
     - 若 review session 前 enabled 一直为 true → 用 `--review-mode-off`(只清 review_mode,无副作用)
     - 若 review session 前已 `--off` 又进 review session → 用 `--on`(双字段都恢复)
     - **保守建议**:不确定时用 `--on`(完整恢复永远 safe;不会让 review_mode 卡 true)
   - `friction --status` 看当前状态(含 `enabled` / `review_mode` R2 B-R2-3 / `schema_error_count` R1 H3)
6. **审阅流程(O4)— R1 H2 + R2 B-R2-3 强警告**:
   - **Step 1 必跑**:`friction --review-mode`(设 review_mode=true,关 hook,防 race;**`enabled` 不变** · R2 B-R2-3)
   - **Step 2**:打开 `docs/dogfood/v4-friction-log.md`,在每条 entry **下一行** 写 `[acked]` / `[disputed]` / `[needs-context]`(可附自由文本)
   - **Step 3 必跑**:编辑保存后 `friction --review-mode-off`(R2 B-R2-3 修订:不再用 `--on`)恢复 hook;hook 在 `enabled=true AND review_mode=false` 时恢复 fire
   - **Why**:hook 与编辑器并发 append 同一文件 → 编辑器 stale snapshot 写回会丢 hook 期间的新 entry + tag 物理位置错位(D5 物理绑定)
   - **Why review_mode 不是 --off(R2 B-R2-3)**:R1 原方案让 `--review-mode` 关 enabled,这会让 day-14 trust_summary `--check-enabled` 在 review 期间 false fail;R2 修订让 review_mode 与 enabled 正交,review session 不影响 adoption verdict(D20)
   - grep 统计 `grep -c "^\[acked\]" docs/dogfood/v4-friction-log.md`
7. **Mid-pilot day-3 / day-7 自检**(TECH-3 + PROD-3 mitigation · R2 M-R2-3 修订:用已实装 CLI 入口替换 health_check 引用):
   ```bash
   # day 3 / 7
   python -m friction_tap.grep --count-agent-emit --since=72h   # 看最近 72h 累计 entry(R2 M-R2-3 替代 health_check)
   python -m friction_tap.trust_summary --check-enabled         # 看 hook 是否还 enabled(R1 B2)
   python -m friction_tap.trust_summary                         # 即跑出 mini report,提前看 ratio
   # 若 disputed > 20% → friction --threshold high
   # 若 entry = 0 且 --check-enabled exit 1 → friction --on(忘起点仪式 · TECH-3)
   # 若 enabled=true 但 entry 仍 0 → 看 friction --status 中 schema_error_count(R1 H3 · 可能 hook payload schema drift)
   ```
8. **Day-14 trust 报告 + self-interview(R1 H1 结构化要求)**:
   - 文件位置 `docs/dogfood/v4-trust-w2.md`
   - 末尾 `**Self-interview**:` prompt 期望 operator 填**两行结构化答案**:
     ```
     [label: <relieved|watched|both>]
     Reason: <一句话 ≥ 20 CJK 字符 OR ≥ 8 English words>
     ```
   - **不接受**:仅一个词("both"),纯空,只填 label 无 reason
   - **怎么验证**:跑 `python -m friction_tap.trust_summary --check-self-interview` → exit 0 表 OK(R1 B2 / O6 verification)
   - **勿当 chore 跳过** — 这是 hybrid 增量唯一 forcing function

8b. **Schema health · 排错(R1 H3 fix)**:
   - day-14 trust 报告 `## Schema health` 段若显示 `schema_error_count: K` 且 K > 0:
     - 看 `~/.config/friction-tap/last_run.log`(尾部 K 行 traceback)
     - 常见原因:Claude Code 升级改了 hook payload 字段名(`tool_name` → `toolName` 等)
     - 修复:更新 friction-tap 到适配最新 Claude Code 的版本;v0.1 期间可手编辑 `tools/friction_tap/judge.py` 与 `hook.py` 解析逻辑兼容
9. **Privacy promise**(指向 docs/privacy.md):一段引言,详见 privacy.md
10. **Uninstall**:`bash scripts/uninstall_launchd.sh` + `friction --off`

### docs/privacy.md 章节
1. **Default private**(C2 + C10):log 文件在 repo 内,默认未 gitignore — operator 自决
2. **Visible commitment**:每行 entry 都含 `[private-to-operator]` 字段 — 不是隐藏假设(C10)
3. **No external network**:hook / CLI / trust_summary 全 offline(C5 RL-3)
4. **No telemetry**:不收集 operator behavior;无 analytics
5. **stderr 200-char cap + D19 best-effort regex redact**(SEC-2 · R2 M-R2-4 修订):**v0.1 实装两层防御** — (a) 200-char cap:hook 写入前对 stderr 截断到 200 字符,降低 secret 落地概率(architecture §5.3);(b) D19 best-effort regex redact:对 7 类常见 token shape 做 regex match 替换为 `[redacted]`(`(sk\|pk\|ghp\|github_pat\|aws\|AKIA)_[A-Za-z0-9]{16,}` / `Bearer [A-Za-z0-9_-]{20,}` 等),细节见 risks.md SEC-2 与 spec.md D19。**两层都是 best-effort**:non-standard token shape 可能漏过(false negative);正常长 identifier 可能被误 redact(false positive,SEC-2 acknowledgment R2 M-R2-4 fix · 是 privacy 优先的 trade)。若 dogfood 期间发现遗漏的 secret 已落 log,operator 仍可手编辑文件删该 entry(D5 markdown 直接编辑)
6. **Repo 移到 iCloud Drive / SMB 不支持**(TECH-5 警告):必须本地 APFS 卷
7. **Multi-user 系统**(SEC-3):state.json 已 chmod 600,但若多人共享机器需额外评估
