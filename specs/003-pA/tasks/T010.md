## Context

**R4 API key exfil + C20 USD 硬帽前置拒绝**(D10 升级 · D-OP1 决策)的核心实现。worker 子进程 env 不持 `ANTHROPIC_API_KEY`,走 localhost proxy;proxy 持 key,HTTPS 回 Anthropic。

**关键升级(对齐 spec v0.2 C20)**:proxy 在 request path 前做 USD 预估 → 余额不足**直接返回 402/429,不发上游请求**。60s 轮询 + SIGINT 降级为第二道保险(T018)。

选 LiteLLM(OQ4 默认),强制 `bind=127.0.0.1`、仅放行 `/v1/messages` 等必要端点、请求日志脱 key 落 `runs/<id>/artifacts/api_log.jsonl`。

关联 Outcome:O5(USD 硬帽 + 计量来源);关联 Risk:R4 / C20。

## Implementation plan

1. 先写 `_pick_free_port`(socket bind on 0 → getsockname)
2. 写 `ProxyConfig` + 校验(非 127.0.0.1 拒绝)
3. 生成临时 LiteLLM yaml config(仅 anthropic provider,key 从 `$ANTHROPIC_API_KEY` 取,显式设 `host: "127.0.0.1"`)
4. `start_proxy`:`subprocess.Popen(["litellm", ...], env={**os.environ})`(proxy 本身持 key,可 inherit env)
5. 健康检查:httpx GET 重试 10 次 × 500ms
6. `audit_log`:用 LiteLLM 的 `success_callback`(支持 custom function),写文件用 T008 风格的 open-append-fsync
7. 集成测试:启 proxy → `subprocess.run(["curl", "--max-time", "2", f"http://{local_ip}:{port}/health"])` 预期 fail(非 127.0.0.1 bind)

## Known gotchas

- LiteLLM 的 `config.yaml` 格式在不同版本间有小变化;pin 到 tech-stack §8 的 litellm 1.50.x
- `_pick_free_port` 有 TOCTOU:bind test 与实际 start 之间端口可能被抢;接受风险(单机单 worker 低概率)
- 若 LiteLLM 启动慢(> 5s)→ OQ4 fallback 改自写极简 proxy(另开 task,不本 task 内切)
- audit log 要定期轮转?v0.1 不做(单 run 量小)
- proxy 的 stderr 可能打印 API key(LiteLLM bug 或 verbose mode)→ 启动时显式 `--quiet` + 重定向 stderr 到 `runs/<id>/artifacts/proxy.err.log` 前过滤
