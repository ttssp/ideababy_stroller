## Context

`metrics.jsonl` 是 worker 训练/eval 子进程 → host 的通信载体(architecture §3 Sequence);也是 report 渲染的数据源(O2)与 compare 的对比基础(O6)。

为避免 worker 崩溃丢数据,必须 **append-only + 每行 fsync**。

关联:O2 训练曲线、O5 wall_clock 来源之一、O6 compare。

## Inputs

- `architecture.md §4.2` metrics.jsonl 语义(append-only,每 epoch / eval 一行)
- 上游:T004 paths, T007 state

## Outputs

- `pars/ledger/metrics.py`:
  - `MetricRecord` Pydantic 模型:
    - `event: Literal["train_epoch","train_step","eval_scores","download_progress","phase_transition","error"]`
    - `timestamp: datetime`
    - `phase: str`
    - `data: dict` — event-specific payload(train_epoch 含 `{epoch, train_loss, eval_loss}`,eval_scores 含 `{task_name, metric, score}`)
  - `append_metric(run_id, record)` — 每次 open-append-fsync-close(抗 crash)
  - `read_metrics(run_id) -> Iterator[MetricRecord]`
  - `aggregate_by_event(run_id, event) -> list[MetricRecord]`
  - `get_training_curve(run_id) -> pd.DataFrame`?**不用 pandas**(压依赖预算);返回 `list[dict]`,matplotlib 直接吃
- `tests/unit/test_metrics.py` — 8+ test:合法 append、格式错误 skip、crash-after-partial-line 处理、并发 append 不交错(flock 或 O_APPEND 原子性依赖)

## Implementation plan

1. `append_metric`:用 `open(path, "a", buffering=1)` + `write(json + "\n")` + `flush()` + `os.fsync(fd)`;在 Linux POSIX 下 `O_APPEND` 本身保证 `write()` < PIPE_BUF 原子性,够用;为跨平台稳妥仍加 fcntl.flock
2. `read_metrics`:yield-based;遇到 JSONDecodeError 记 log warning + skip 该行(rollback 时可能有半行)
3. 测试覆盖:
   - 写 100 行,读回来 100 行顺序一致
   - 注入一行 `{malformed...` 到文件中间,`read_metrics` skip 该行且继续
   - 两个线程各 append 50 条,最终 100 条、无行破损

## Known gotchas

- `O_APPEND` 在 Linux 保证 < PIPE_BUF (4KB) 写的原子性,本项目单条 metric record 远小于此;加 fcntl.flock 是防御性双层
- **不**引入 pandas;下游 report 用 list[dict] + matplotlib 原生 ok
- 不做 metric 压缩 / 归档(v0.1 单 run 最多几千行)
