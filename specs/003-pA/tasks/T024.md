## Outputs

- `pars/orch/retry.py`:
  - `derive_retry_config(parent_run_id, hypothesis, overrides: dict) -> RunConfig`
    - 读 parent config
    - 应用 hypothesis 到 next_steps(parent 的 failure_attribution)建议的 overrides
    - 或直接 apply CLI 提供的 `--lr` / `--lora-rank` 等 overrides
    - 新 run_id(新 ULID)
    - `config.parent_run_id = parent_run_id`(schema 字段加一个)
    - `config.retry_hypothesis = hypothesis`
- `pars/cli/retry.py`:
  - `pars sft retry --hypothesis TEXT --from RUN_ID [--lr FLOAT] [--lora-rank INT] ...`
  - 调 derive → 新 orchestrator.start
- `tests/unit/test_retry_config_diff.py` — 覆盖 config 派生逻辑:覆盖 / 保留 / parent_run_id 正确
- `tests/integration/test_retry_from_run.py` — 跑一个 parent run + retry,断言新 run 有 parent_run_id 字段 + overrides 生效

## Implementation plan

1. `RunConfig` schema 新增 `parent_run_id: str | None` + `retry_hypothesis: str | None` 字段(T006 schema 需要 patch)— 此改动在本 task 做,不回去动 T006,因为向后兼容(新字段默认 None)
2. `derive_retry_config`:读 parent → copy → override(优先级:CLI override > hypothesis 自动建议 > parent)
3. hypothesis 自动建议逻辑简单:
   - `"LR 太高"` → `lr /= 3`
   - `"LR 太低"` → `lr *= 3`
   - `"epoch 不足"` → `epochs += 2`
   - `"rank 不足"` → `lora_rank *= 2`
   - 其他 → 不自动改,仅记录 hypothesis
4. CLI 命令:薄 wrapper 调 derive + orchestrator.start

## Known gotchas

- 需要改 T006 config schema(加字段),但属向后兼容(Optional 字段,默认 None);T006 已合并的旧 config 仍可读
- hypothesis 自动建议是启发式,用户可能不满意 → 永远允许 CLI 显式覆盖
- parent run 不存在 → 退出码 2 + 提示
