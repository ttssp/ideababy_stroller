## Outputs

- `pars/budget/tracker.py`:
  - `class BudgetTracker`:
    - `usd_cap: float`, `wall_clock_cap_s: float`, `gpu_hours_cap: float`
    - `tick(run_id) -> BudgetStatus` — 每 60s:**读 `state.json.usd_spent`**(T010 proxy 已原子更新,无需再解析 api_log);读 state.started_at → 算 wall-clock;读 gpu_hours state → 累加
    - `check_caps(status) -> list[str]` — 返回被突破的项(empty 即 OK)
    - 写 `state.json`:更新 `wall_clock_elapsed_s / gpu_hours_used`(USD 不由本 task 写,避免冲突)
  - `class BudgetMonitor(Thread)`:
    - 每 60s 跑 tick;若 wall-clock / GPU hours cap 突破 → `worker.stop(SIGINT)` + log fatal
    - 若 USD 通过此路径被触发 → log **warning "usd cap hit via 60s polling fallback, proxy pre-reject may have missed"** + SIGINT(预期正常情况下不应走到这里,若频繁触发表明 T010 proxy 预估有漏洞)
- `tests/unit/test_budget_tracker.py` — 10+ test(模拟 state.usd_spent → 核对;模拟时钟跳 > cap → 标记 over)
- `tests/integration/test_budget_sigint_on_cap.py` — 模拟 mock worker 跑到 **wall-clock** 或 **gpu_hours** over → Monitor 发 SIGINT(USD 路径由 T010 `test_proxy_prerejects_on_budget.py` 覆盖,本 task 不重复)

## Implementation plan

1. **不再自行维护 pricing.py**:从 `pars.proxy.budget_estimator` import `MODEL_PRICING` / `compute_usd`(T010 拥有单一真相源)
2. `BudgetTracker.tick`:
   - 读 **state.json.usd_spent**(T010 proxy 权威值)
   - `wall_clock = now - state.started_at`
   - 写回 state.json(仅 wall_clock / gpu_hours 字段,不覆盖 USD)
3. `BudgetMonitor`:复用 T017 的监控循环模式(单独 Thread,60s 间隔)
4. cap 突破逻辑:简单比较,不用 Decimal(USD 精度 2 位小数够);wall_clock 用 seconds;gpu_hours 估算可能不准,接受
5. 测试:用 `tmp_runs_dir` + 合成 state.json(mock `usd_spent`),调 `tick` 看兜底行为

## Known gotchas

- **pricing.py 归 T010 owns**;本 task 从 `pars.proxy.budget_estimator` import(避免定价二义)
- Anthropic 定价会变;每季度人工 review 一次;README 里说明"若定价变了,改 `pars/proxy/budget_estimator.py`"
- `state.usd_spent` 权威来源 = T010 proxy;本 task 读不写该字段(避免 race condition)
- wall-clock 精度受 60s 轮询限制;但 SLA §1.3 明示 < 5s 延迟是目标,wall-clock 可用 state.started_at + now() 精确计算
- GPU hours 近似实现:`gpu_util>5% 的 60s tick 数 × 60 / 3600`;可靠性不高,但 v0.1 接受("近似"在 README 声明)
