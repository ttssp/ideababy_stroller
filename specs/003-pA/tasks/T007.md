## Context

两个小而关键的底层模块:
- ULID 生成(**D14 决策**:时间可排序 + 唯一 + 可读),`pars sft start` 自动给每 run 发一个
- `state.json` 读写(resume 的载体,**C13** 硬约束),含阶段 / checkpoint 路径 / 已用 budget / stuck 状态

关联:O3 resume(C13)、O5 budget、O7 stuck。

## Outputs

- `pars/ledger/run_id.py`:
  - `generate_ulid() -> str` — 用 `ulid-py` 生成 26 字符 Crockford Base32
  - `validate_ulid(s) -> bool`
  - `parse_ulid_timestamp(s) -> datetime` — 便于 `pars status` 显示创建时间
- `pars/ledger/state.py`:
  - `RunState` Pydantic 模型,字段:
    - `run_id: str`
    - `phase: Literal["init","baseline","lora_train","eval","report","completed","failed"]`
    - `current_script: str | None` — `baseline_script.py` / `lora_script.py` / `eval_script.py`
    - `checkpoint_path: str | None` — 绝对路径
    - `last_epoch_completed: int | None`
    - `stuck_state: Literal["idle","training","downloading","truly_stuck"]` — 默认 "idle"
    - `stuck_restart_count: int` — 默认 0
    - `needs_human_review: bool` — 默认 False(熔断信号)
    - `usd_spent: float`, `wall_clock_elapsed_s: float`, `gpu_hours_used: float`
    - `last_update: datetime`
  - `read_state(run_id) -> RunState`
  - `write_state(run_id, state)` — atomic
  - `update_state(run_id, **kwargs)` — flock + 部分更新
- `tests/unit/test_run_id.py` — 5+ test
- `tests/unit/test_state.py` — 10+ test(含 phase 流转合法性)

## Implementation plan

1. `run_id.py`:包装 `ulid-py`,提供稳定 API(未来若切 `python-ulid` 不影响调用方)
2. `state.py`:参考 T006 `config_io` 的 atomic + flock 模式,复用策略;**不**共享代码(此 task 要先于 T006 就能跑,保持自足)— 如果 T006 已合并,此 task 可 refactor 共享 `_atomic_write_with_lock`,放入 `pars/ledger/_io.py`
3. phase 流转合法性:定义 `VALID_TRANSITIONS: dict[str, set[str]]`,`update_state` 检查;拒绝非法 transition 时抛 `ValueError`(如从 `completed` 回到 `lora_train`)
4. 测试:hypothesis 不装,用参数化 pytest 覆盖所有合法/非法 transition

## Known gotchas

- `ulid-py` 1.1 与 Python 3.12 兼容性确认过;如果装不上,fallback `python-ulid`
- `datetime` 用 `datetime.now(timezone.utc)`(UTC,CLAUDE.md 一般规则)
- phase 流转允许 `* → failed` 与 `lora_train → lora_train`(resume 情形)— 显式枚举
