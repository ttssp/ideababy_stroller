## Context

**R3 Critical(LLM 失败归因质量差)的核心缓解 = D16 半结构化 schema**。若 worker 交"可能是数据问题,可能是超参问题"产品价值崩塌;schema 强制必填字段 + 枚举归因类 + 必引用具体 metric。

本 task 定义 schema + 写给 worker 的 prompt 模板 + 校验器。

关联 Risk:R3(Critical);关联 Outcome:O2(报告含失败归因)。

- `PRD.md §9` 归因质量 biggest risk

- 上游:T004 paths,T006 config(eval metric 字段名)

## Outputs

- `pars/report/failure_schema.py`:
  - `FailureAttribution` Pydantic 模型:
    - `assumption: str`(必填,≥ 20 chars)
    - `observation: str`(必填,≥ 30 chars,**必须**引用 metric 数字;实现用正则 `r"\d+\.\d+"` 检测存在数字)
    - `causes: list[CauseCategory]`(必填,≥ 1 项)
    - `causes_detail: str`(可空,若 CauseCategory.OTHER 必填)
    - `next_steps: list[str]`(必填,≥ 1 项,每项包含具体 hyperparam / 数据 / 模型 change)
    - `free_narrative: str | None`(可空,但鼓励)
  - `class CauseCategory(Enum)`: 对齐 architecture §9 枚举(DATA_FORMAT, LR_TOO_HIGH, LR_TOO_LOW, DIST_DRIFT, BASELINE_STRONG, EPOCH_NOT_ENOUGH, LORA_RANK_LOW, OOM, OTHER)
  - `parse_markdown(md_text: str) -> FailureAttribution` — 解析 worker 写的 `failure_attribution.md`(section + checkbox 解析)
  - `validate_quality(fa: FailureAttribution) -> tuple[bool, list[str]]` — 二次校验(除 Pydantic 外的语义:observation 含数字、next_steps 至少含 1 个具体词如 `lr=`/`epoch=`/`data=`)
  - **Contract 统一(spec v0.2 §6 O2 + R1 Medium #2 修复)**:validate_quality 在**生产** schema 严格 pass(False 即 report schema gate 失败);消费端(T021 / T026 schema validator)也严格 gate — 不存在"警告但不 fail"的半态语义,消除 T020/T021 漂移。
- `pars/report/failure_prompt.py`:
  - `FAILURE_PROMPT_TEMPLATE: str`(Jinja2)— 给 worker 的 prompt 指示,含:
    - 必填字段列表 + 示例
    - 禁用词黑名单(`可能是` / `也许` / `大概` / `不确定`;若 observation 含则二次要求)
    - 必须引用 metric 数字的明确指示
    - 归因枚举列表(对齐 CauseCategory)
  - `render_failure_prompt(run_id, baseline_score, lora_final_score) -> str`
- `tests/unit/test_failure_schema.py` — ≥ 15 test:
  - 合法 FA → ok
  - observation 无数字 → `validate_quality` fail
  - next_steps 无具体词(全是"继续尝试")→ fail
  - causes 空 → Pydantic error
  - OTHER 但 causes_detail 空 → fail
  - parse_markdown happy path
  - parse_markdown 缺 section → ValueError

## Implementation plan

1. `FailureAttribution` 用 Pydantic v2 + `field_validator`
2. `parse_markdown`:按 H2 section 切分(`## 必填字段` / `## 自由叙述`),用正则抓 `- **假设**:` 等字段
3. `validate_quality`:正则 `r"\b\d+\.?\d*\b"` 检测数字;`next_steps` 每项检 `r"(lr|epoch|rank|batch|data)\s*=\s*\S+"` 或至少 1 处数字变更
4. `FAILURE_PROMPT_TEMPLATE`:中文为主(worker 也是中文),给 3 个示例(好的 + 不好的 + 边缘)
5. 测试:16 个 test,每条都附 comment 说明防哪种崩塌模式

## Known gotchas

- `observation 含数字` 的正则要小心:"epoch 3" 也算数字;接受 — 至少证明 worker 引用了可观测事实
- `next_steps 含具体变更` 的正则:`lr=1e-4` OK;`try lower lr` 也算 OK(含 `lr`);完全模糊"继续尝试"不含任何关键字 → fail
- prompt 模板用中文,对齐 CLAUDE.md 偏好;模板示例段要展示"bad → good"对比防崩塌模式
- **validate_quality 严格 gate**(R1 Medium #2 决策):若 worker 忘写 metric 数字或 next_steps 模糊 → **schema fail**,report 不通过 O2 gate;操作员需手工补写或让 worker 重跑;避免"报告漂亮但内容空"(PRD §9 崩塌模式)
