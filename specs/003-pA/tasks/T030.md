class FleetDispatcher:
    """编排多个 RunOrchestrator,按 GPU lock 决定启动时机."""
    
    def __init__(self, batch: BatchSpec, locks_root: Path): ...
    
    def run(self) -> list[RunHandle]:
        """阻塞跑 batch,返回所有 RunHandle.
           核心 loop:
             pending = list(batch.runs)
             active = []   # (BatchRunSpec, RunHandle, GpuLock, Thread)
             while pending or active:
               for run_spec in pending:
                   target_gpu = pick_gpu(run_spec, batch.gpu_strategy)
                   lock = GpuLock(target_gpu, locks_root)
                   if lock.acquire(run_spec.name, timeout_s=0):
                       handle = launch_in_thread(RunOrchestrator.start, run_spec_to_config(run_spec))
                       active.append((run_spec, handle, lock, thread))
                       pending.remove(run_spec)
               # 5s tick:检查 active 谁完了 → release lock
               sleep(5)
               for run_spec, handle, lock, thread in active[:]:
                   if not thread.is_alive():
                       lock.release()
                       active.remove(...)
        """
    
    def cancel(self):
        """Ctrl+C handler:不 kill workers,只退出 dispatcher loop;
           workers 自己有 stuck/budget monitor 接管(R15)"""

## Known gotchas

- 多线程 dispatcher loop 与 GIL:每个 RunOrchestrator.start 是阻塞调用 → 用 threading.Thread 包(I/O bound,GIL OK)
- KeyboardInterrupt 在多线程下不会自动传递:主线程捕获 → dispatcher.cancel() → 标志位 → 子线程下次 tick 看到标志 → 退出 loop;**子线程不强 join**(它们是 daemon thread,程序退出自动 reap)
- pick_gpu 算法:`auto` = 选第一个 lock 未持有的 gpu;`manual` = 严格按 run_spec.gpu_uuid;两种都失败 → wait
- depends_on 字段虽然 C25 warn+ignore,Schema 仍要接受(extra="forbid" 不报错)
- fleet_name 从 batch 注入到每个 RunConfig → v0.1 RunConfig 不知道这个字段?**需要在 RunConfig 加 optional fleet_name: str | None = None**(scope 外微调,与 T024 增加 retry_hypothesis 同模式)
