## Goal

写 solo operator 日常运维手册（runbook）与 O4 周度 self-report 模板。Runbook 覆盖 "每日巡检 5 分钟"、"月度审计 15 分钟"（compliance §7）、"incident 响应流程"。Self-report 模板落 `briefings/self-report/` 目录，operator 每周日手工填一份，包含 O1/O2 数字 + "若无本工具会漏看"截图 slot。纯 prose 任务，haiku 合适。

## Inputs

- `specs/001-pA/spec.md` §6 O4（self-report + 截图证据）、§6 Sentinel 验收
- `specs/001-pA/compliance.md` §7（月度自审清单）
- `specs/001-pA/risks.md` OPS-1/OPS-2/TECH-4/SEC-3（incident 场景）
- `specs/001-pA/SLA.md` §1.5 / §1.6（incident response + 可观测性）
- T030 产出：systemd service 名、pg_dump 路径

## Outputs

- `docs/runbook/README.md` — 索引：日常 / incident / 自审 / self-report 各指向的子文档
- `docs/runbook/daily-operations.md` — 每日/每周巡检表：
  - 每日：`systemctl status pi-briefing-*` / `journalctl -u pi-briefing-worker | grep -E 'ok|failed'`
  - 每周：self-report + `/admin/sentinel` / `llm_calls` 累计查询
  - 每月：compliance §7 的 6 条自审清单
- `docs/runbook/incident-response.md` — 按常见事故分章节：
  - "briefing 没生成"（TECH-2）→ 查 fetch_runs → 手工重跑 `pnpm worker:daily`
  - "LLM cost 超 $40"（C11 + SLA §1.6）→ 查 llm_calls → 降级为 heuristic-only
  - "VPS 失联"（OPS-3）→ restore drill 流程（链到 T031 `restore-drill.md`）
  - "export 异常"（SEC-1）→ audit log 查 requester + 撤销
- `docs/runbook/self-report.md` — 空模板（weekly）：
  - 本周 active seats（从 /admin/sentinel 查）
  - 本周 4-action 数量（从 SQL 查）
  - "若没有本工具会漏看"案例 ≥ 1 个（附截图路径 / 文件名）
  - day-45+ 时：breadcrumb aha 例（O5）
- `briefings/self-report/README.md` — 说明：每周日创建 `YYYY-WW.md`（ISO 周号）；文件名示例 `2026-18.md`
- `briefings/.gitkeep` — 保留目录结构；`briefings/self-report/*.md` 默认 git-tracked（operator 选择）但 `briefings/*.png` screenshots 可选

## Implementation plan

1. 四份 markdown 文档，按上面概要撰写；用中文。
2. 用表格 + 代码块 + 编号清单；保持在每份 ≤ 150 行，避免变成墓碑
3. self-report 模板给 6–8 个 fillable 字段，格式统一（方便未来 O1-O5 SQL aggregation）
4. incident-response 里每条事故必须给：**症状**、**快查命令**、**缓解步骤**、**事后要做的事**
5. runbook README 作为单页索引，operator bookmark 即可访问所有子页

## Known gotchas

- Runbook 要贴近 operator 实际生活：不写"最佳实践"空话，给具体命令
- 避免 "TODO" 遗留；任何"待 v0.2 补"的 marked 为 `> ⚠️ v0.2 预留`
- self-report 不走 git mandatory，但强烈推荐（operator 的承诺物证）；`briefings/` 目录要 git-tracked，**不**把 screenshot 自动 push（包含潜在 PII）
- 所有 SQL 查询命令要在 runbook 里附 expected output 样例
- incident-response 的每一条都要能在 5 分钟内操作完（不要写成技术博客）

## Out of scope

- 不写 spec 级别产品决策变更流程（那是 /spec-from-conclusion）
- 不写 CLAUDE.md / AGENTS.md（CLAUDE.md 明确禁止自动生成）
- 不建 wiki / external docs site（boring over clever）
- 不做 video tutorial
- 不做 API 文档（v0.1 无外部 API 消费者）
