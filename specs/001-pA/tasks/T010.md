## Goal

实现 topic 管理：admin 可 CRUD，member 只读。**硬守 SLA §4 红线 1**：topic 总数 > 15 时 POST 返回 400（"不扩成通用发现器"机械兜底）。Topic 字段 = {name, keyword_pool: string[], arxiv_categories: string[], seed_authors: string[]}（spec IN-1）。UI 刻意粗糙（tech-stack.md §1.2、PRD UX principle 1）：表格 + 表单，无组件库。

## Inputs

- `specs/001-pA/spec.md` IN-1（topic 定义）、C5 红线 1
- `specs/001-pA/SLA.md` §4（R1 机械兜底：`count(topics) < 15`）
- `specs/001-pA/architecture.md` §5.1（`topics` schema：name/keyword_pool/arxiv_categories/seed_authors/archived_at）
- T003 产出：`topics` schema + `db` client
- T006 产出：`requireAuth` / `requireAdmin` middleware

## Outputs

- `src/lib/topics/validation.ts` — zod schema：name 2–80 char、keyword_pool 1–20 项 每项 ≤ 40 char、arxiv_categories 1–5 项 matching `^[a-z-]+\.[A-Z]{2}(-[A-Z]{2})?$`（如 `cs.CL`）、seed_authors 0–10 项
- `src/lib/topics/service.ts` — 导出 `listTopics()`, `getTopic(id)`, `createTopic(input)`, `updateTopic(id, input)`, `archiveTopic(id)`；createTopic 内做 `SELECT count(*) FROM topics WHERE archived_at IS NULL` >= 15 时 throw `TooManyTopicsError`（**G4 H2 · 2026-04-24 权威 error code `TOO_MANY_TOPICS` + HTTP 422**，与 `reference/api-contracts.md §E6` + `error-codes-and-glossary.md §1` 一致；旧 `TopicCapError` / 400 已废弃）
- `src/app/(main)/topics/page.tsx` — 列表页（server component）：展示 topic 表格；**新建是模态或内联表单**（Server Action），不走独立页面；admin 看到 "新建" 按钮与 "编辑"/"归档"
- `src/app/(main)/topics/[id]/edit/page.tsx` — 编辑单条 topic 的独立页（**G4 H2 · 路径权威 `[id]/edit/page.tsx`**，与 `directory-layout.md §1` 一致；旧 `[id]/page.tsx` 已废弃）
- `src/app/api/topics/route.ts` — GET（list）、POST（create, requireAdmin）
- `src/app/api/topics/[id]/route.ts` — GET、PATCH（update, requireAdmin）、DELETE（实际上 archive, requireAdmin）
- `tests/lib/topics/service.test.ts` — 所有 CRUD + cap edge case（15 → 16 被拒 with `TOO_MANY_TOPICS`）
- `tests/app/topics/route.test.ts` — HTTP 层：非 admin POST 返 **403 `NOT_ADMIN`**、> 15 topic POST 返 **422 `TOO_MANY_TOPICS`**（G4 H2）

## Implementation plan

1. `validation.ts` 用 zod；导出 `CreateTopicInput` / `UpdateTopicInput` 类型。
2. `service.ts`：所有 DB 操作用 T003 `db` client；`createTopic` 先 cap check，再 INSERT；`archiveTopic` = UPDATE `archived_at = now()` 而不是 DELETE（保留历史）；cap 违反 throw `class TooManyTopicsError extends Error`（handler 映射 HTTP 422 `TOO_MANY_TOPICS`）。
3. Admin 判断从 middleware 已注入的 `seats.role`；service 层不重复 auth（保持单一真源）。
4. 页面 server components：直接 `await service.listTopics()`；新建表单通过列表页的内联或模态表单（Server Action），**不**走独立 `/topics/new` 路径；编辑走 `/topics/[id]/edit`（G4 H2）。
5. 错误处理：cap 超时 → handler 返回 **422** + `{ error: { code: 'TOO_MANY_TOPICS', message: 'topic 数已达 15 上限，请先归档不再关注的 topic' } }`（G4 H2 · 与 api-contracts.md §E6 一致）。
6. 不引入 shadcn / radix（tech-stack.md §1.2）；用 tailwind + 原生 `<table>` / `<input>` / `<button>`。

## Known gotchas

- topic cap check 有并发竞争：两个 admin 同时 POST 第 15 条可能都通过。v0.1 acceptable（≤ 15 seat + ≤ 3 admin，实际不会并发），用 advisory lock 太重；在 service 中加显式 note 留给 v1.0 解决
- arXiv category 正则涵盖 `cs.CL`, `stat.ML`, `cs.LG-NN`（破折号变体），不要过严拒 valid category
- keyword_pool 允许中文；`min length = 1`，避免误提交空串
- `archived_at` 的 topic 不再参与 daily fetch worker 匹配（T011 默认只查 `WHERE archived_at IS NULL`）
- UI 要明确展示 "当前 / 15" topic 数（操作员要知道剩余 quota）

## Out of scope

- 不做 per-topic ACL（OUT-10 + NG-A10）
- 不做 topic 关系图可视化（OUT-2）
- 不做 topic 推荐 / 自动补充 keyword（会扩成通用发现器，违反 R1）
- 不做 admin panel 的 seat CRUD（NG-B4）
