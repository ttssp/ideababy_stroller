## Goal

落地 SLA §1.3 双层备份：本地 `pg_dump`（T030 已做）+ **每周 restic/rclone 副本到 operator 云盘**。同时写 `restore-drill.md` 运维剧本，并在一次性脚本 `restore-drill.sh` 验证 RTO ≤ 2h（从备份拷回 → pg_restore → systemctl restart → 页面可访问）。本 task 是 TECH-4 / OPS-3 的直接 mitigation，也是 C12/C8 数据主权承诺的兜底。

## Inputs

- `specs/001-pA/SLA.md` §1.3（RPO ≤ 24h、RTO ≤ 2h）
- `specs/001-pA/risks.md` TECH-4（pg_dump + restic 云盘）、OPS-3（VPS 失联时 ≤ 4h 换机）、BUS-2（源码丢失应对）
- `specs/001-pA/compliance.md` §4（数据主权 invariants：备份不离 operator 掌控）
- T030 产出：`/var/backups/pg/` 每日 dump 存在

## Outputs

- `deploy/backup/restic-sync.sh` — 初次 `restic init -r $REPO`；后续 `restic backup /var/backups/pg/ --tag weekly`；`restic forget --keep-weekly 12`
- `deploy/backup/restic-sync.service` — Type=oneshot；ExecStart 调 restic-sync.sh；EnvironmentFile 含 RESTIC_REPOSITORY, RESTIC_PASSWORD_FILE
- `deploy/backup/restic-sync.timer` — `OnCalendar=Sun *-*-* 07:00:00 Asia/Shanghai`；Persistent=true；每周日一次
- `deploy/backup/restore-drill.md` — 运维剧本：
  1. ssh 新 VPS（或现 VPS）
  2. 如源码丢：`git clone` + provision
  3. `restic restore latest --target /tmp/backup-restored`
  4. `pg_restore -d pi_briefing -U postgres /tmp/backup-restored/var/backups/pg/latest.dump`
  5. `systemctl start pi-briefing-web pi-briefing-worker.timer`
  6. curl /today 确认返回 200
  7. 记录实际耗时到 self-report
- `scripts/restore-drill.sh` — 自动化 drill：在 VPS 上 `pg_restore` 到 `pi_briefing_drill` 临时 DB，`psql` 校验 row count vs. primary DB，drop 临时 DB；完整 drill 目标 < 30 min（远低于 RTO 2h）
- `tests/ops/restore-drill.test.sh` — bash 测试：mock 一个 dump → 执行 scripts/restore-drill.sh → 断言 row count 一致

## Implementation plan

1. restic：选 operator 掌控的 S3/Backblaze B2/SFTP backend；env `RESTIC_REPOSITORY` 形如 `b2:bucket:/pi-briefing`；密码走 `RESTIC_PASSWORD_FILE=/etc/pi-briefing/credentials/restic-password`
2. 脚本首次运行 `restic init` 幂等（若已 init 返回非 0 容错）
3. drill：为避免影响生产 DB，restore 到 `pi_briefing_drill` schema；完成后 `DROP DATABASE pi_briefing_drill`
4. `restore-drill.md` 写中文 step-by-step；operator 每季度跑一次实际 drill 确认 RTO
5. `restic forget --keep-weekly 12` 保留 3 个月；加 `prune` 在每月第一周
6. log 要告诉 operator："本周备份完成；仓库使用 X GiB；远端最新 tag 是 Y"

## Verification (MUST be runnable)

- [ ] 在 VPS 首次执行 `bash deploy/backup/restic-sync.sh` 成功退出码 0；远端仓库可见对象
- [ ] `systemctl list-timers` 包含 restic-sync.timer
- [ ] `bash scripts/restore-drill.sh` 执行 ≤ 30 min，退出码 0，`pi_briefing_drill` 已 DROP
- [ ] 手工 RTO 测试：关停 web/worker → drop DB → restore from restic → 启动 → /today 200；全程记录耗时（目标 < 2h，实际 ≤ 30 min）
- [ ] `restic snapshots -r $REPO` 列出 ≥ 1 snapshot
- [ ] `restore-drill.md` 被 T033 runbook 链接到
- [ ] `tests/ops/restore-drill.test.sh` 在 CI 或 local 一次性 smoke 通过（mock pg_dump input 文件）

## Known gotchas

- restic 的 password-file 权限必须 600，且 systemd service 的 user（pi-briefing）有读权限
- B2 限速：首次同步 1GB 需 ~3 分钟；每周备份通常 < 200MB 增量
- `pg_restore` 对新机器的 role 要求：`webapp_user` / `worker_user` 必须先存在（T030 init.sql 保证）
- Drill 时要锁定 "临时 DB"，避免 operator 误 drop 真 DB；脚本里 `set -u` + check DATABASE_NAME=drill 前缀
- RTO 包含"代码 + 数据 + 服务重启"：代码从 GitHub pull（BUS-2 覆盖）、数据从 restic、服务从 systemctl
- 云盘 provider 失败路径：若 B2 outage，operator 手动 scp `/var/backups/pg/` 到其他位置；v0.1 不做自动二级 fallback

## Out of scope

- 不做多地冗余备份（v0.1 一份 local + 一份 restic 够）
- 不做 WAL archiving（streaming replication）；v1.0 再考虑
- 不做自动故障转移（NG-B1 + solo operator）
- 不做 email 报告备份结果（operator 看 journalctl 即可）
