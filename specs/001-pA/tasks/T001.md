## Goal

为整个项目的 LLM provider 选型拍板。operator 先人工标注 20 篇 arXiv paper（标签 = state shift / incremental），随后用同一份 fixture 对 Claude Sonnet 4.6 与 GPT-5.4 做盲测，比较 `state-shift judgment 准确率`、`summary ≤ 3 句质量`、`token 成本`、`p95 latency`。退出条件来自 tech-stack.md §2.5：至少一家 ≥ 70% 准确率 + 月度成本 ≤ $50 + p95 ≤ 5s；若两家都 < 70%，fallback 到 §4.1 纯启发式 + LLM 仅做 summary，同时在 spec.md OP-Q1 上打标。此任务是 Phase 1 的 **blocking gate**，operator 必须在报告里 commit `approved-provider: <name>` 一行，Phase 1 才允许开始（spec.md §6 T001 spike 额外验收）。

## Implementation plan

0. **Collect fixture**：operator 选 20 篇近 30 天的 AI/ML arXiv paper，分布为 **10 篇已知 shift**（Mamba、DPO、GPT-4 report 等公认引发方向变化的工作）、**6 篇 incremental**（同 topic 但仅小幅改进）、**4 篇 unrelated**（跨 topic 干扰项）；逐条人工打标并写到 `tests/fixtures/human-labeled-20.json`，严格匹配 Outputs 里的 Fixture shape（`candidate` / `anchor` / `topic` / `humanVerdict` / `rationale` 五字段缺一不可；`anchor` 允许 null 当 candidate 与任一锚点都不构成 shift 关系）。
1. operator 自选 20 篇近 30 天的 AI/ML arXiv paper。要求 ~10 篇 shift（有前置 anchor + ≥ 2 篇引用且立场冲突）、~10 篇 incremental（同 topic 但无 state 变化）。每条手工写人类标签与简短 note。
2. 写 `eval-harness.ts`：CLI 接收 `--provider claude|gpt`，读 fixture，对每篇调用 `summarize(title, abstract)` 与 `judgeRelation(anchor, candidates)`（直接调 SDK，**不用** T004 的 adapter 避免循环依赖）。
3. 采集指标：tokens（in/out）、latency（per-call ms）、summary 句数（`split /[.!?]\s+/`）、judgeRelation label 与人类标签一致性。
4. 运行两家各 1 次，结果写入 `runs/*.json`；若任一单次请求失败 exponential-backoff retry ≤ 3 次。
5. 计算准确率、成本外推（按 tech-stack.md §2.1/§2.2 pricing × fixture 放大到月度）、p95 latency。
6. 写 `T001-llm-provider-report.md`：逐条对比 + 推荐理由 + `approved-provider: <name>` 占位行等 operator 签字。
7. operator 追加签字行 + git commit。

## Known gotchas

- arXiv API 有 rate limit（≤ 1 req/3s）；fixture 采集脚本需尊重 sleep
- 两家 SDK 的 `temperature` / `max_tokens` 缺省值不同；harness 必须显式设定同一数字（建议 `temperature=0.2, max_tokens=300`）以保证公平比较
- OpenAI `gpt-5.4` 的结构化输出用 `response_format: { type: 'json_object' }`；Anthropic 用 `tool_use` 强制；harness 代码两端都要处理
- 本任务 output 到 `projects/001-pA/spikes/`，**不进** `src/`；不影响后续 build worker 的 file_domain
- `risks.md` TECH-1 明确"若 < 70%"要 flag spec.md OP-Q1，不要在 report 里擅自选 provider
- 若 T004 LLMProvider interface 尚未实现，本 spike 直接调 SDK（不绕 T004 adapter），避免循环依赖；产物是报告 + fixture，无生产代码

## Out of scope

- 不写 T004 的 LLMProvider interface（那是 T004 的事）；spike 直接调 SDK 更快
- 不跑 local LLM 评测（tech-stack.md §2.3 明确 v0.1 不在 T001 评估）
- 不做 Phase 1 briefing 预计算或 state-shift pipeline（spike 只做 fixture 级测量）
