## Goal

为整个项目的 LLM provider 选型拍板。operator 先人工标注 20 篇 arXiv paper（标签 = state shift / incremental），随后用同一份 fixture 对 Claude Sonnet 4.6 与 GPT-5.4 做盲测，比较 `state-shift judgment 准确率`、`summary ≤ 3 句质量`、`token 成本`、`p95 latency`。退出条件来自 tech-stack.md §2.5：至少一家 ≥ 70% 准确率 + 月度成本 ≤ $50 + p95 ≤ 5s；若两家都 < 70%，fallback 到 §4.1 纯启发式 + LLM 仅做 summary，同时在 spec.md OP-Q1 上打标。此任务是 Phase 1 的 **blocking gate**，operator 必须在报告里 commit `approved-provider: <name>` 一行，Phase 1 才允许开始（spec.md §6 T001 spike 额外验收）。

## Implementation plan

0. **Collect fixture**：operator 选 20 篇近 30 天的 AI/ML arXiv paper，分布为 **10 篇已知 shift**（Mamba、DPO、GPT-4 report 等公认引发方向变化的工作）、**6 篇 incremental**（同 topic 但仅小幅改进）、**4 篇 unrelated**（跨 topic 干扰项）；逐条人工打标并写到 `tests/fixtures/human-labeled-20.json`，严格匹配 Outputs 里的 Fixture shape（`candidate` / `anchor` / `topic` / `humanVerdict` / `rationale` 五字段缺一不可；`anchor` 允许 null 当 candidate 与任一锚点都不构成 shift 关系）。
1. operator 自选 20 篇近 30 天的 AI/ML arXiv paper。要求 ~10 篇 shift（有前置 anchor + ≥ 2 篇引用且立场冲突）、~10 篇 incremental（同 topic 但无 state 变化）。每条手工写人类标签与简短 note。
2. 写 `eval-harness.ts`：CLI 接收 `--provider claude|gpt`，读 fixture，对每篇调用 `summarize(title, abstract)` 与 `judgeRelation(anchor, candidates)`（直接调 SDK，**不用** T004 的 adapter 避免循环依赖）。
3. 采集指标：tokens（in/out）、latency（per-call ms）、summary 句数（`split /[.!?]\s+/`）、judgeRelation label 与人类标签一致性。
4. 运行两家各 1 次，结果写入 `runs/*.json`；若任一单次请求失败 exponential-backoff retry ≤ 3 次。
5. 计算准确率、成本外推（按 tech-stack.md §2.1/§2.2 pricing × fixture 放大到月度）、p95 latency。
6. 写 `T001-llm-provider-report.md`：逐条对比 + 推荐理由 + `approved-provider: <name>` 占位行等 operator 签字。
7. operator 追加签字行 + git commit。

## Known gotchas

- arXiv API 有 rate limit（≤ 1 req/3s）；fixture 采集脚本需尊重 sleep
- 两家 SDK 的 `temperature` / `max_tokens` 缺省值不同；harness 必须显式设定同一数字（建议 `temperature=0.2, max_tokens=300`）以保证公平比较
- OpenAI `gpt-5.4` 的结构化输出用 `response_format: { type: 'json_object' }`；Anthropic 用 `tool_use` 强制；harness 代码两端都要处理
- 本任务 output 到 `projects/001-pA/spikes/`，**不进** `src/`；不影响后续 build worker 的 file_domain
- `risks.md` TECH-1 明确"若 < 70%"要 flag spec.md OP-Q1，不要在 report 里擅自选 provider
- 若 T004 LLMProvider interface 尚未实现，本 spike 直接调 SDK（不绕 T004 adapter），避免循环依赖；产物是报告 + fixture，无生产代码

## Out of scope

- 不写 T004 的 LLMProvider interface（那是 T004 的事）；spike 直接调 SDK 更快
- 不跑 local LLM 评测（tech-stack.md §2.3 明确 v0.1 不在 T001 评估）
- 不做 Phase 1 briefing 预计算或 state-shift pipeline（spike 只做 fixture 级测量）
- 不把 `fixtures/labeled-20.json` 的 paper 推入 Postgres（spike 独立于 DB，避免 Phase 0 schema 未就绪时阻塞）

## Deviations

> **Deviations 摘要**：T001 v0.1 在两处偏离原契约，均已在 2026-04-24 operator review 后接受并同步到 harness + fixture 实现。若未来 revision 需恢复原契约（双 provider 对比 + arxivId-only fixture），应先 revert 本段两条 deviation。

### D2 · v0.1 scope reduction to single GLM provider (2026-04-24)

- **原契约**：`--provider claude|gpt`，Phase 0 gate 要求 Anthropic Sonnet 4.6 + OpenAI GPT-5.4 双家盲测对比，得出 `approved-provider: anthropic | openai | FALLBACK_TO_HEURISTIC`
- **当前执行**：`--provider glm`，走 `@anthropic-ai/sdk` + 火山 ARK Anthropic-compatible endpoint（`ANTHROPIC_BASE_URL=https://ark.cn-beijing.volces.com/api/coding`），只跑火山引擎 GLM5.1 单家
- **理由**：operator 当前无 Anthropic / OpenAI API key，买 key 会让 Phase 0 卡住；GLM5.1 通过同一 Anthropic 协议，可作为 v0.1 Phase 1 准入 gate 的最小可行对接
- **Exit 条件不变**：T001 gate 仍然要求 **judge_accuracy ≥ 14/20**（即 ≥ 70%）；若 GLM5.1 达不到，走 spec.md OP-Q1 fallback-to-heuristic 路径（LLM 仅做 summary）；若通过，`approved-provider: glm-5.1` 由 operator commit 追加
- **已对齐文件**：`projects/001-pA/spikes/eval-harness.ts`、`.env.example` LLM 段（`ANTHROPIC_BASE_URL` + `ANTHROPIC_API_KEY` 指向火山 ARK）
- **未来 scope**：当 operator 拿到 Anthropic / OpenAI API key 后，在**独立 revision** 内重开 T001 双 provider 对比 spike；本次 v0.1 不等
- **对下游 task 的影响**：T004 adapter 仍按双 provider interface 实现（`llm-adapter-skeleton.md §3/§4` 契约不变），只是 `src/lib/llm/select.ts` v0.1 只会真正实例化 Anthropic adapter（endpoint 指向 GLM）；Claude / GPT 分支保留代码路径但 v0.1 不激活

### D3 · anchor.paperId integer field added to fixture shape (2026-04-24)

- **原契约 Fixture shape**：`anchor: { arxivId, title, abstract }`
- **新契约 Fixture shape**：`anchor: { arxivId, paperId: integer, title, abstract }`（新增 `paperId` 整数字段）
- **理由**：`llm-adapter-skeleton.md §3` 的 `VerdictSchema.anchorPaperId` 定义为 `z.number().int().nonnegative()` —— 整数；harness 必须向 judge 传整数 `anchor_paper_id` 才能让 LLM 返回 Zod-valid JSON。原设计用 `candidate.arxivId`（字符串）占位会让 judge 返字符串，Zod 拒绝 → 所有 shift 判决退化为 ZodError，gate 永远无法过
- **paperId 取值**：v0.1 spike 内用 fixture 数组索引（f1→1, f2→2, …, f20→20）作占位整数；T003 DB schema 落地后由 operator 用真 `paper_id` 替换
- **已对齐文件**：`tests/fixtures/human-labeled-20.json` 的 `_meta.d3_patch` 字段 · `projects/001-pA/spikes/eval-harness.ts` 的 `FixtureAnchor` 类型 + `renderJudgeUser` 实现
- **对下游 task 的影响**：T012 state-shift pass 实现时，anchor 的真 `paper_id` 由 DB query 得到（非 fixture）；v0.1 spike 内的占位 paperId **只在 T001 评测范围内有效**，不得被 T012 复用
