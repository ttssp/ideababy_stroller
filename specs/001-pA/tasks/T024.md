## Goal

实现 **DOGFOOD-1 mitigation**（risks.md）+ spec §6 Sentinel 验收：day-30 起自动计算 `active_seats_30d` = 过去 30 天有 ≥ 1 次 login 的 seat 数；若 < 3 → `/today` 顶部显示红色 escalate banner "触发 escalate，请 operator 人工复盘" 并可读 `allow-continue` flag（operator 填入数据库置位后 banner 消失）；若 < 2（day-60 视角）→ banner 阻塞新 feature 开发（Phase 2 sentinel 要求）。这是 v0.1 **non-negotiable** 的 kill-window guardrail。

## Implementation plan

1. `active-seats.ts`：`SELECT COUNT(DISTINCT seat_id) FROM sessions WHERE login_date >= current_date - $1::int`
2. `banner.ts`：读 `labs.first_day_at`（T003 seeds 时填 first_day_at=now）；`since_day = differenceInDays(now, first_day_at)`
3. `sentinel-banner.tsx` server 端直接 `await getSentinelStatus()`；无 JS state
4. `/admin/sentinel` 页面：admin 专用；展示 sentinel metric + 按钮 "我已人工复盘，允许继续运行 7 天"（POST 设 allow_continue_until = now+7d）；之后 banner 在 7 天内不显示
5. day-60 阻塞"新 feature": v0.1 没有自助 feature flagging，spec 的意思是 banner 明写 "应回 L2 重新 scope"——操作员看到自己决定。此任务把文案落实即可，不强制禁 UI 其他 feature。
6. worker log 扩展：`runBriefingPass` 末尾 `console.log('[SENTINEL]', { active_seats_30d, active_seats_60d })`；operator `journalctl -u pi-briefing-worker | grep SENTINEL` 可快速审视

## Known gotchas

- **R1 前置闭合（B3）**：`labs.first_day_at` + `labs.allow_continue_until` 已由 T003 R1 前置；本 task 直接读列
- `since_day = 30` 作为触发起点严格按 spec §6 "day-30 时自动计算" 意思是 since_day >= 30 就开始评估；不要少 1 天
- banner 不是 `"use client"`；用 server component 避免 hydration 不一致
- operator "allow-continue" 机制需要 7 天 TTL：避免 operator 临时忽略后忘了回头检查；每 7 天必须再签一次
- 本 task 仅展示 banner 与 admin 页；**不阻塞** CRUD feature（例如 topic 不能因 sentinel 失败而拒绝新增）；SLA §4 的红线机械兜底也不通过 sentinel 触发，sentinel 是建议性（risks.md DOGFOOD-1 是 "阻塞新 feature 开发" 的操作指引，不是代码层阻塞）
- **R1 co-ownership（B3）**：`/today/page.tsx` 由 T015 owner、T022 也 co-own（放 ResurfaceBanner），本 task 放 SentinelBanner 在最顶部；三者按 `depends_on: [T015, T022]` 串行
