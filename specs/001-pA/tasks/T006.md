## Goal

落地 architecture.md §7 的完整 auth 骨架：email token invite（24h 有效、单次使用、hash 存库）→ magic-link verify → JWT httpOnly session cookie → middleware 拦截所有非 `/login*` 路由。PI + operator 各自独立 login（spec IN-5 + D3），admin 与 member 角色分离（SEC-1）。**所有路由默认 deny**，白名单 `/login`、`/login/verify` 通过后才可访问其他页面。这是 C7 / OUT-8 红线 3 的机械守门。

## Outputs

- `src/lib/auth/session.ts` — `createSession(seat_id) / verifySession(token) / revokeSession(id)` 用 `jose` 签名 JWT；30 天滑动过期；`sessions.token_hash` 存 sha256(token) 供 revocation
- `src/lib/auth/invite.ts` — `createInviteToken(email, role)` 生成 32-byte random → hash 存 `sessions` 变体表**不**（注：invite token 不存 sessions；存入 `seats.invite_token_hash` 临时列）；导出 `consumeInvite(token)` 单次使用后 clear hash
- `src/lib/auth/email.ts` — 用 `nodemailer` 发 magic link 邮件；subject `[pi-briefing] Your invite to ...`；不含其他 user PII
- `src/lib/auth/middleware.ts` — 导出 `requireAuth(request)` / `requireAdmin(request)`；抛 `AuthError('unauthorized')` / `AuthError('forbidden')`；副作用写 `sessions.last_active_at` 滑动续期
- `src/middleware.ts` — Next.js 15 middleware 在 edge 拦截；白名单 `/login`, `/login/verify`, `/api/healthz`；其余 302 到 `/login`
- `src/app/login/page.tsx` — email 输入 + "我没收到邀请 → 联系 lab admin" 注释；POST 到 `/login` route handler 触发 "该 email 存在则发 magic link"（列出存在性不泄漏——always 200）
- `src/app/login/verify/route.ts` — GET `?token=xxx`：consumeInvite 成功 → 新 session cookie → 302 `/today`；失败 → 302 `/login?error=expired`
- `src/app/api/auth/invite/route.ts` — POST（requireAdmin）：`{email, role}` → createInviteToken + email；返回 202
- `src/app/api/auth/logout/route.ts` — POST：revokeSession + clear cookie
- `tests/lib/auth/session.test.ts` — JWT sign/verify roundtrip、expired token rejected、revoked token rejected
- `tests/lib/auth/invite.test.ts` — 24h 过期、单次使用、wrong token rejected
- `tests/lib/auth/middleware.test.ts` — requireAuth 对无 cookie 请求返 401；requireAdmin 对 member role 返 403

## Implementation plan

1. `session.ts`：用 `jose.SignJWT` + `HS256` 签名；payload `{ sub: seat_id, iat, exp }`；`verifySession` 校验 `exp` 与 `sessions.revoked_at IS NULL AND sessions.token_hash = sha256(token)`。
2. `invite.ts`：`crypto.randomBytes(32).toString('base64url')`；`seats.invite_token_hash` 列存 `sha256` + `seats.invite_expires_at`；`consumeInvite` 事务里校验并置 null。
3. **数据库 schema（R1→R2 Q2 resolution · 2026-04-23）**：T003 已前置 ship `seats.invite_token_hash` / `seats.invite_expires_at` / `seats.invited_at` / `sessions.revoked_at` / `sessions.last_active_at` 五列（随 `0000_initial.sql` 一次 ship，见 T003 Implementation 步骤 14）。**T006 不再写任何 migration 文件**，只从 `src/db/schema.ts` import 类型 + 从 `src/db/client.ts` 拿到 `db` / `dbWorker` 执行读写。
4. `middleware.ts`：匹配 `matcher: ['/((?!login|_next/static|_next/image|favicon.ico|api/healthz).*)']`；无 cookie → `NextResponse.redirect('/login')`。
5. 邮件 magic link `{APP_ORIGIN}/login/verify?token=<plain>`；plain token 仅出现在 outbound 邮件 body；日志里只写 hash（SEC-2）。
6. `sessions.login_date` 在 session 创建时写当日本地日期（Asia/Shanghai），供 O1/O3 index 查询。
7. 所有 handler 捕获 throw，返回统一 `{ error: 'unauthorized' | 'forbidden' }`，**不暴露** "email not found" vs "wrong token" 差异（SEC-2）。

## Known gotchas

- Next 15 middleware 运行在 edge runtime；`jose` ok，但 `crypto.randomBytes` 只能在 node runtime——`invite.ts` 用 Web Crypto `crypto.getRandomValues`
- `sessions.login_date` 必须用 `Asia/Shanghai` 时区截日，否则 O1 跨零点统计会错 1 天
- Nodemailer SMTP 失败时不要把 "邮件没发出" 返回给前端（SEC-2：避免暴露 "该 email 是否存在"）；fallback 是 log + 后端告警
- `jose` HMAC secret 要用 `new TextEncoder().encode(SESSION_SECRET)` 且 secret ≥ 32 byte
- **R1→R2 Q2 resolution（2026-04-23）**：T003 已 ship 这 5 列（`seats.{invite_token_hash, invite_expires_at, invited_at}` + `sessions.{revoked_at, last_active_at}`），T006 只写 `src/lib/auth/invite.ts` + session middleware，不再需要写 migration。若本地 merge 顺序异常（T003 尚未 apply），先 rebase 拉 T003 的 schema；**禁止** 在 T006 PR 里新增 `src/db/migrations/0002_auth_columns.sql`
