## Goal

实现 **PI 每日 8:00 ritual 的主页 `/today`**：SSR 读 `briefings` + `paper_summaries`（JOIN per-paper summary 文本）+ trigger paper 详情（title/abstract），无 LLM 调用在请求路径（ADR-2），p95 < 1.0s（SLA §1.2）。每个 paper 旁边是 4 个 action 按钮（read_now / read_later / skip / breadcrumb）+ 一个可选的 `why` 文本输入（≤ 280 字符，spec IN-3）。action POST 到 `/api/actions` 或 Server Action；写入 `actions` 表，同时当 action=breadcrumb 时**同步写 `breadcrumbs` 表**（architecture §5.1 约定）。

**R1 核心变更（B1 + B2）**：
- **B1**：summary 文本不再从 `llm_calls` 取；统一从 `paper_summaries(paper_id, topic_id)` JOIN；没有 summary（T013 fallback 失败或 T001 LLM disabled）时，UI 显示占位 `"（LLM 暂不可用，查看原始 abstract）"`
- **B2**：红线 2 "skip traceable" 三层机械兜底的第 2 + 第 3 层：
  - **API 层**（`recordAction.ts`）：action='skip' 且 why 为空或 btrim < 5 字符 → 返回 400 `{error: 'skip requires why (≥5 chars)', code: 'SKIP_WHY_REQUIRED'}`，不触发 DB 写
  - **UI 层**（`skip-why-input.tsx`）：点击 skip 按钮**不**立即 POST；先 inline 展开一个 `<textarea placeholder="why skipping this? (required, ≥5 chars)">`，实时计数，按钮 disabled 直到文本 ≥ 5 字符（trim 后）才可 commit

## Implementation plan

1. `loader.ts`：一次 SQL 拉完 topics + 其 briefing + trigger paper + **paper_summaries（LEFT JOIN）** + 当前 seat 的 my_action（用 LEFT JOIN `actions` ON seat_id = ? AND paper_id = triggered paper）。
   ```sql
   -- 粗略示意：
   SELECT b.topic_id, t.name AS topic_name, b.state_summary, b.anchor_paper_id,
          p.id AS paper_id, p.title, p.abstract,
          ps.summary_text, ps.model_name AS summary_model,
          a.action AS my_action, a.why AS my_why
   FROM briefings b
   JOIN topics t ON b.topic_id = t.id
   LEFT JOIN papers p ON p.id = ANY(b.trigger_paper_ids)
   LEFT JOIN paper_summaries ps ON ps.paper_id = p.id AND ps.topic_id = b.topic_id
   LEFT JOIN actions a ON a.paper_id = p.id AND a.seat_id = $seat
   WHERE b.lab_id = $lab AND b.date = $date
   ORDER BY t.name, p.id;
   ```
   注意：`paper_summaries` 可能有多个 prompt_version —— loader 取 latest（按 created_at DESC + LIMIT 1）；latest prompt_version 由 `src/lib/llm/prompt-version.ts` 的常量决定。

   **summary 选择规则**（权威 SQL 片段，loader 内部按 paper/topic 逐条选）：
   ```sql
   SELECT ...
   FROM paper_summaries
   WHERE paper_id = ? AND topic_id = ?
   ORDER BY prompt_version DESC, created_at DESC
   LIMIT 1
   ```
   优先最新 `prompt_version`；相同 `prompt_version` 取最新写入（`created_at DESC`）。
2. `recordAction.ts`（**R1 新增 API 层 B2 验证** · Result-return 模式，与 skeleton `specs/001-pA/reference/skeletons/recordAction.ts` 对齐 —— **不 throw，直接 return** `{ ok: false, code, message }`）：
   ```ts
   // zod schema 定义 skip-requires-why ≥5 chars + why ≤280 chars；parsed = ActionInputSchema.safeParse(raw)
   export async function recordAction(raw: unknown): Promise<RecordActionResult> {
     const parsed = ActionInputSchema.safeParse(raw);
     if (!parsed.success) {
       const issue = parsed.error.issues[0];
       if (issue?.path[0] === 'why') {
         return { ok: false, code: 'SKIP_WHY_REQUIRED', message: 'skip 必须填写 why (≥ 5 chars)', fields: { why: 'at least 5 chars required' } };
       }
       return { ok: false, code: 'INVALID_ACTION', message: parsed.error.message };
     }
     // ... 后续 paper 存在性 / duplicate / breadcrumb 事务写入，见 skeleton
     return service.recordAction(parsed.data); // { ok: true, actionId, breadcrumbId? }
   }
   ```
   **API Route 层映射**（`src/app/api/actions/route.ts`）：该层负责把 Result 翻成 HTTP envelope —— `ok: false` → HTTP 400 + JSON `{error: message, code, fields?}`；`ok: true` → HTTP 200 + JSON `{ok: true, ...}`。DB CHECK `skip_requires_why` 是 Layer 1 最后兜底；zod refine 是 Layer 2 主防线。见 `api-contracts.md` §1 error envelope。
3. `service.ts`：事务内；`if (action === 'breadcrumb')` → 查 `paper_topic_scores` 拿任一 topic_id → INSERT breadcrumbs。若 paper 没有 topic 匹配（边界情况），fallback topic_id = null? 不行——`breadcrumbs.topic_id` 非空；此时应在 loader 阶段就不展示该 paper，service 再 defensive check throw。DB CHECK `skip_requires_why` 是最后一道防线，正常流 API 层已拦截。
4. `action-buttons.tsx`：`"use client"` + `useTransition`；4 按钮 onClick；**skip 按钮**不立即 submit，而是切换 `<SkipWhyInput>` 展开；其它 3 个 action 可直接提交（why 可选）。read_later / breadcrumb 的 why 文本走一个可选的 `<textarea maxlength=280>` 字数计数器。
5. `skip-why-input.tsx`（**R1 新增 UI 层 B2 兜底**）：
   ```tsx
   'use client';
   export function SkipWhyInput({ paperId, onSubmit }: Props) {
     const [why, setWhy] = useState('');
     const trimmed = why.trim();
     const valid = trimmed.length >= 5;
     return (
       <div className="inline-expand">
         <textarea
           value={why}
           onChange={(e) => setWhy(e.target.value)}
           maxLength={280}
           placeholder="why skipping this? (required, ≥5 chars)"
         />
         <div className="chars">{trimmed.length} / 5 min · 280 max</div>
         <button disabled={!valid} onClick={() => onSubmit(trimmed)}>
           Submit skip
         </button>
       </div>
     );
   }
   ```
6. `/today/page.tsx`：server component（async），顶部 `<Suspense>` + heading `"今日 briefing · YYYY-MM-DD"`；按 topic 分块展示。每个 topic 块：`<h2>topic.name</h2>` + `<p>state_summary</p>` + anchor paper（如有）灰色展示 + triggers 列表（每条含 `summary_text` or 占位）+ ActionButtons。
7. 样式：tailwind utility；无组件库；列表使用 `<article>`/`<section>`；保持 viewport ≥ 1024 的单栏 layout（OUT-5）。
8. p95 保障：loader 的 SQL 一次性聚合，少 round-trip；browser 无 client-side fetching（除非 action）。

## Known gotchas

- briefings 是 per-lab per-topic per-date；loader 不能把多个 lab 的数据混读（v0.1 单 lab 部署，但多 lab SQL 仍要 scope by lab_id 防未来回归）
- 当日未生成 briefing（worker 失败或 catchup 中），页面要 graceful 显示 "生成中，请稍后" 而非 500
- breadcrumb 同步写 breadcrumbs 表：必须在同事务，否则 T021 resurface scheduler 会漏扫
- `actions` 表可重复：同 seat 同 paper 可以改主意从 read_later 变 breadcrumb（业务允许）；**不去重**，审计完整
- why 字段的 trim：前后空格去除，但中间保留；空串视为 null
- **R1 重要 gotcha（B1）**：paper summary 从 `paper_summaries` 读，**不**再从 `llm_calls` 读。如果 T001 走 fallback 路径或 T013 provider 全挂，`paper_summaries` 里仍会有 `model_name='fallback-heuristic-v1'` 的行（T013 Known gotchas 约定）；UI 需在此情况下显示 summary 来源 tag，例如 `⚠️ heuristic fallback`（非强制，operator v0.1 可裸展示文本）
- **R1 重要 gotcha（B2 / 在 T001 失败路径上 summary 可能为 null）**：若 `paper_summaries` 完全没有行（极端情况：T013 全挂 + heuristic fallback 也失败），UI 必须 graceful 占位 `"（LLM 暂不可用，查看下方原始 abstract）"`，不让页面 500
- **R1 新增（B2）**：skip UI 不用 modal（modal 会把 10 分钟 ritual 打断）；用 inline expand 在 paper card 内部
- **R1 新增（B2）**：`recordAction.ts` 的 `HTTPError` 类型需要 `code` 字段（`SKIP_WHY_REQUIRED` / `WHY_TOO_LONG`）以便前端 i18n / 错误区分
- UI 占位文案当 `paper_summaries` 为空（T001 fallback 路径）：显示 `(summary 暂不可用 · LLM 未运行)` 灰色斜体；见 skeleton `today-page-client.tsx` 的 `SummaryCell` 组件
