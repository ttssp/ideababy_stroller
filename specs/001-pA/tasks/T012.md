## Goal

落地 spec §4.1 的 state-shift v0.1 正式定义：对每个 topic T，遍历过去 7 天新 paper 集合 `P_new(T)`；找"被 ≥ 2 篇 p_i ∈ P_new 引用且引用中至少一对有 conflicting label（supports vs contradicts/supersedes）"的 anchor earlier paper `e`；输出 `{topic_id, trigger_papers, anchor_paper, label_conflict}`。支持 **降级模式**：若 T001 spike 判定 LLM 准确率 < 70%，则跳过 LLM judge，**shift = 有任一 anchor 被 ≥ 2 篇引用**（heuristic-only），spec §4.1 明写 "保留 heuristic 可 adjust 阈值"。

## Outputs

- `src/lib/state-shift/types.ts` — 导出 `StateShiftCandidate`, `StateShiftResult`, `AnchorEvidence` 类型
- `src/lib/state-shift/anchor-finder.ts` — 导出 `findAnchorCandidates(topic_id, windowDays = 7)`：纯 SQL 查询 `SELECT cited_arxiv_id, array_agg(citing_paper_id) FROM paper_citations JOIN papers ON ... WHERE published_at > now() - interval '7 days' AND topic matches GROUP BY cited_arxiv_id HAVING count(DISTINCT citing_paper_id) >= 2`
- `src/lib/state-shift/engine.ts` — 导出 `runStateShiftPass(topic_id, provider?)`：
  - call `findAnchorCandidates`
  - 若 `env.LLM_JUDGE_ENABLED === 'true'` 且 provider 提供 → call `provider.judgeRelation({ anchor, candidates })` 获取 labels
  - 检查 labels 中是否有 conflicting pair (supports + contradicts/supersedes)
  - 满足 → 生成 `StateShiftResult` 并返回；不写库（由 T014 briefing precompute 读）
  - fallback 模式（LLM disabled）：anchor 被 ≥ 2 引用即视为 shift，`label_conflict` 留空
- `tests/fixtures/state-shift/` — 3 个 topic 的 fixture：(a) 有 shift + conflicting labels、(b) 有 anchor 但标签都 supports（非 shift）、(c) 无 anchor 被 ≥ 2 引用（frontier stable）
- `tests/lib/state-shift/anchor-finder.test.ts` — SQL 正确性；边界（只 1 引用、自引用）
- `tests/lib/state-shift/engine.test.ts` — LLM 启用 vs 禁用两条路径；mock provider 返回不同 label 组合断言 shift 判定

## Implementation plan

1. `anchor-finder.ts`：SQL 用 Drizzle 表达式；JOIN `paper_citations` → `papers`（`papers.arxiv_id = paper_citations.cited_arxiv_id`）→ `paper_topic_scores` 过滤 topic。**注意**：可能存在被引 paper 不在 `papers` 表（外部老论文）——这种情况只拿 `cited_arxiv_id` 字符串作 anchor key，LLM 没法 judgeRelation（缺 abstract），fallback 到 heuristic-only 对这类 anchor。
2. `engine.ts`：构造 `{anchor: { title, abstract }, candidates: [{ id, title, abstract }]}` 传给 provider；对同一 anchor 的 2 paper pair-wise 检查 label 冲突。
3. 定义 "conflicting label"：存在 candidate A label=supports AND candidate B label=contradicts OR supersedes；`unrelated` 不算 shift。
4. Fallback：`if (!env.LLM_JUDGE_ENABLED) { return candidates.map(c => ({ anchor, triggers: c, label_conflict: null })) }`；等于完全信 heuristic。
5. 引入 `LLM_JUDGE_ENABLED` 到 T007 env schema（追加）；默认 `false`（**0.4.0 修订**：Phase 0/1 全程默认 false 走 heuristic-only · Phase 2 T001-v2 通过后由 operator 手工翻 `true`）。
6. 所有 DB 读走 `dbWorker`（worker_user）客户端（架构 §7 双用户分离）。
7. 单元测试用 `withTempSchema` 建临时表、插 fixture、跑 engine。

## Known gotchas

- spec §4.1 "不强行捏造"：若 topic 7 天内无 shift，`runStateShiftPass` 返回空数组，**不**在 briefing 里硬塞"今日无 shift"之外的内容。briefing 展示交给 T014
- TECH-5：worker 日志里要记录每 topic 的 anchor 候选数与 shift 数；operator 可以在 week-2 review 阈值（≥ 2 是否太松）
- LLM judge 一次最多 candidates=5（超过就分 batch）——tech-stack.md §2.4 未设上限但成本考虑
- `paper_citations.cited_arxiv_id` 可能是 `papers` 表未收录的 arxiv_id（旧论文），join 需要 `LEFT JOIN`；fallback 模式也能正确处理
- "冲突对"判定逻辑要清晰：**任意一对**冲突即 shift；不要要求所有 pair 都冲突
