## Goal
决策档案 post-mortem 回填 (architecture §3.2 PostMortem 字段, v0.1 由 human 手动填):
- 决策详情页提供 "回填事后回报" form: `executed_at` (若 wait/hold 留空), `result_pct_after_7d`,
  `result_pct_after_30d`, `retrospective_notes`
- 提交时调 `DecisionRepository.update_post_mortem(trade_id, payload)`
- 列表页加列 "post_mortem 状态" (待回填 / 已回填); 自动 nudge: 决策 7 天 + 30 天后
  在 weekly review 列出 "本周需回填的决策"
- "不动 / 等待" 决策 (action ∈ {hold, wait}) 也允许 post_mortem (R11: 不动是正式输出,
  允许复盘)

## Inputs
- T002 `PostMortem` 模型
- T003 `DecisionRepository.update_post_mortem`
- T008 决策详情页 (本 task 加 sub-route)

## Outputs
- `services/post_mortem_service.py`: 校验 (executed_at 必须 ≥ pre_commit_at;
  result_pct_after_7d 仅在 7 天后允许填)
- `ui/router_post_mortem.py`: GET `/decisions/{trade_id}/post-mortem`, POST 同 path
- 模板: form 含 4 字段, 默认填 (若已回填一次)
- 单测: post_mortem 时间逻辑校验
- 集成测: 创建决策 → POST post-mortem → 详情页显示已回填

## Implementation plan
1. service 校验 + 调 repo
2. router GET 渲染表单, POST 入库
3. 模板: 简单 form, hold/wait 不显示 executed_at
4. 单测覆盖时间约束
5. 集成测试
