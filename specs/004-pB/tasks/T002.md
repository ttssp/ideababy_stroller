## Goal
把 architecture.md §3.1 的 IDL 数据类与 §3.2 决策档案 SQLite DDL 落成 pydantic 2 模型 +
SQLite schema (含所有补充表: `advisor_reports`, `parse_failures`, `watchlist`, `holdings_snapshot`,
`conflict_reports`, `rebuttals`, `notes`, `alerts`, `llm_usage`, `pending_jobs`, `meta_decisions`,
`request_log`, `errors`, **R2 新增 `decision_drafts`**)。配置 alembic, 写第一份 migration `0001_initial_schema.py`,
覆盖架构 §3.2 的全部 DDL + §3.1 IDL 字段所需表 + 索引 + WAL pragma 设置。该 task 是
所有后续业务任务的依赖底座。

## Implementation plan
1. 写 8 个 domain 模块, 每个文件单职责
2. pydantic validators: action enum、reason length、rationale_plain 非空、
   conflict_report signals ≥ 3
3. 写 alembic env.py 让 url 来自 config.py
4. 写 0001_initial_schema.py 用 raw SQL `op.execute(...)` 创建表 (不引入 SQLAlchemy ORM,
   配合 tech-stack §1 决策)
5. alembic env.py 启动时执行 `PRAGMA journal_mode=WAL; PRAGMA foreign_keys=ON;`
6. 写 round-trip 单测: 临时 sqlite 上跑 `alembic upgrade head` → `alembic downgrade base`
   → `alembic upgrade head` 全部成功
7. 在 migration 前增加 `pre-migration backup` hook (env.py 内复制 db.sqlite 到 backups/)

## Known gotchas
- pydantic v2 与 v1 的 validator 装饰器不同, 用 `@field_validator`
- `would_have_acted_without_agent` 在 SQLite 用 INTEGER 0/1 存, pydantic 用 bool
- migration 文件本身要可逆, 即使 v0.1 永远不 downgrade (TECH-8)
- 不引入 SQLAlchemy ORM (tech-stack §1 决策), 仅用 raw SQL DDL
