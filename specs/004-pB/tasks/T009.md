## Goal
实现 O5 验证 (R3 修订口径, spec.md §6 O5, 守 PRD §S1 原口径):
- (a) **自动化 E2E timing test** (Playwright Python binding): 模拟键盘输入完整双阶段流程, 度量**全程 wall-clock < 30s** (从 GET /decisions/new 200 OK 起算)
  - 阶段 1 (draft 创建): 测 ≤ 5s (R3 D21 上限)
  - 阶段 2 (commit): 测 < 1s (无 LLM)
  - **全程 (合): < 30s** (C11 硬门槛, R3 守 PRD 原口径)
- (b) **手动压测脚本** `scripts/manual_press_test.py`: 引导 human 连续 5 次手工录入, 程序记录每次**全程 wall-clock** + 阶段分解 (draft / commit), 全部全程 < 30s 才写入 `release_log.jsonl`; 任何一次 ≥ 30s 标 fail

测试在 ship 前 + 每次 release 必跑. failing → release block (PRD R1 硬门槛, OP-1 触发降级 B-lite 候选).

## Known gotchas
- Playwright headless 模式比真浏览器快, 阈值留余量 (CI 设 < 25s 比 30s 安全, 真用仍 < 30)
- 中文输入法对录入耗时影响显著 — manual 必须包含中文 reason 真实键入
- CI runner 性能波动, 阈值用 P95 不是单次
- **R3 修订**: draft 阶段 LLM 等待**包含在 30s 全程内** (R3 守 PRD §S1 原口径); 计时点严格 = GET /decisions/new 200 OK 起算
- **R3 修订**: 测试需覆盖 cache 命中 (≤ 1s) / cache miss (≤ 5s) / 超时 (≥ 6s → 503) 三种 draft 场景
