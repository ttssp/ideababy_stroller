## Goal
个人金融笔记 wiki (D14 nav 第 4 项): human 写关于关键概念的解释 (e.g. "P/E vs P/B" /
"FOMC 分点投票"). 系统职责:
- CRUD: 笔记可创建/编辑/删除
- 自动去重: 新笔记 content_hash (T002 schema) 检查 + LLM 语义去重 (note_dedup
  prompt: "已有笔记 X 与新笔记 Y 是否解释同一概念?"); 命中提示"已有相同笔记"
- agent 引用笔记: 当 LLM 生成 conflict_resolve / weekly_review 时, 若引用一个概念,
  优先链接到 human 的笔记 而不是再解释 (减少噪声; OP-7 防信号黑箱要求白话, 但避免
  agent 重复造词)
- 全文搜索 (SQLite FTS5)

## Inputs
- T002 `Note` 模型 (content_hash 字段)
- T003 `NoteRepository.find_by_content_hash`, `search_fulltext`
- T005 LLMClient (note_dedup prompt 调用)

## Outputs
- `services/note_service.py`: CRUD + dedup check
- `ui/router_notes.py`: GET `/notes`, GET `/notes/new`, POST `/notes`,
  GET/POST `/notes/{id}/edit`, DELETE `/notes/{id}`, GET `/notes?q=keyword`
- 模板: 列表 (按 updated_at DESC) + 编辑表单 + 卡片 partial
- `note_dedup_v1.md`: prompt 模板, 输出 `{is_duplicate: bool, existing_note_id:
  string | null, reason: str}` (Anthropic tool use)
- alembic migration `0003_notes_fts.py`: 增加 SQLite FTS5 virtual table 同步触发器
- 单测: hash 命中 + LLM 语义命中 + 搜索 hit
- 集成测: 创建笔记 → 创建相似笔记 → UI 显示 dedup 提示
- 测试: agent 调用时 (e.g. weekly_review) 优先 link 而非重新解释 (mock 验证 prompt
  上下文中插入笔记 link 列表)

## Implementation plan
1. service: hash dedup 同步; LLM 语义 dedup 异步 (在保存前 prompt 一次, 命中提示
  human 是否合并 / 链接旧 note)
2. router CRUD + search
3. FTS5 migration (002 + new 003)
4. 模板用 Pico CSS, 笔记卡片简洁
5. 单测: hash collision case + 非命中 case
6. 集成测: TestClient 创建 → 创建近似 → 命中 dedup
7. agent 引用笔记的接口: `note_service.list_summaries_for_topics(topics: list[str])
  -> list[NoteRef]`, T010 / T014 / T015 / T019 调用 (本 task 提供, 上述各 task 各自决定
  是否调用)

## Known gotchas
- LLM dedup 是软提示, 不强制 (OP-7 不冒充判官; human 决定合并)
- FTS5 中文分词差; 用 Trigram tokenizer 改善
- content_hash 只对 normalize 后内容算 (lower + strip)
