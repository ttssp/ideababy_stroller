## 0. Alternative framings considered

在落到「PRD → 自动开发管道」这个解读之前，我想过另外几种读法。

**(A) 「我要一个永远比我懂软件的搭档」**——把 005 读成"配一个高级技术合伙人型 agent"，重点不在管道而在持续陪伴。这个读法把人放在每一步循环里：human 提需求，agent 解释取舍，human 拍板，agent 实施。但 human 写得很清楚——"几乎没有人工干预"、"自动化程度最高"。所以这是被否决的读法。如果有一天 human 收回"自动化优先"，这个读法会复活。

**(B) 「我要一套可复用的工程纪律 skills 库」**——把 005 读成"做出比 addy osmani 的 agent-skills + superpowers 更系统的 skills 集合"，重点是产出复用品。这个读法不错，但太工具化——human 已经在 autodev_pipe 里走了这条路（"借鉴最佳实践，做 pipeline"），结果他还是觉得不够、又开了第四个 repo。所以纯 skills 库不是终极目标，skills 是"为了让管道更可靠"的手段。被降级为 §4 的扩展。

**(C) 「我要让 claude code 自己写 PRD、自己开发」**——从一句话需求一路全自动到产品。诱人但越界：human 自己说"我可以将需求描述清楚，我也可以尝试构建较可靠的 PRD"，明确把 PRD 划在自己的责任区。把 PRD 也吃掉就是不尊重 human 的现有能力分工，且把不确定性堆在最危险的需求侧。本 L2 不走这条。

**(D) 「我要做的其实是一个 idea-incubator-as-a-product」**——把当前 repo 的工作流（Proposal→L1→L2→L3→L4）打磨成对外可用的产品。这是把工作流商品化。诱人，但会模糊 005 的真问题——005 不是"对外卖工作流"，是"我个人怎么把 PRD 可靠地变成软件"。这个读法可以是 L2 后的一个 fork（"如果 005 v0.1 work 了，第二年可以考虑产品化"），但不是 L2 主线。

落点：005 的本质是**「PRD-driven 全自动开发管道」**——把 human 已经能产出的 PRD，变成 agent 几乎独立完成的可信软件。下面按这个解读展开。

## 1. The idea, unpacked

**用户画像，要具体**：005 的目标用户不是泛泛的"非程序员"，而是一类很特别的人——**有产品判断力、能写出结构化需求、对工程纪律有审美但缺执行经验的"产品型创造者"**。human 自己就是样本：CS PhD 背景、ML 出身、能讲清楚"为什么要 TDD"和"什么是 spec-driven"，但从没在大公司跟过几个版本的迭代节奏，没踩过"测试覆盖率假高真低"的坑，不知道"灰度发布"的真实意义。这群人有钱有时间有品味，但缺一种深度：**"把一个想法跨过工程实现的鸿沟到达可信赖的软件"这件事的肌肉记忆**。

**005 之前的日常**：human 现在每开一个新项目都要重做一次同样的事——挑 skills、挑 subagents、写 playbook、跑一段、回头改 prompt、再跑、发现 agent 偷工减料、加约束、回归测、走 loop。每个项目都是从零搭脚手架。idea_gamma2 是一种搭法、vibe-workflow 是另一种、autodev_pipe 是第三种、当前 repo 又是第四种。**重复的不是想法，是工程纪律的迁移成本**。每次都在重新发明"怎么让 agent 更可靠"。

**005 之后的日常**：human 写完 PRD 之后，把它丢进管道，离开屏幕去做别的事；几小时（或几天）后回来，看到的是一个有完整测试、有变更说明、有遗留问题清单、有 review 反馈、有可灰度部署的产物。**关键不是"它写完了代码"，是"它的产物 human 敢相信"**——敢部署、敢给别人看、敢在它基础上继续迭代。中间过程 human 不需要每一步都批，但所有重要决策点（架构选型、scope 取舍、测试策略）都有可审计的"agent 当时为什么这么决定"的记录。

**「aha」的 30 秒**：是这样一个时刻——human 写完 PRD 第一次跑管道，一两个小时后回来看，发现管道不仅产出了代码，还**主动提出了 PRD 里没写清楚的两个边界问题**（"你说要导出 CSV，但没说大于 100MB 的怎么处理；我做了 streaming 方案+一个降级到分片下载的 fallback，标了 TODO 等你拍"）。这一刻 human 意识到：管道不是个执行体，是个**有判断的合作者**。它没等他补需求就自己做了合理选择，并且把决策点暴露出来。这种"我没说但它考虑到了"的体验是 vibe coding 的反面——vibe coding 是"它想到了但你没跟上"，005 是"它想到了并按你的工程标准帮你处理好了"。

**6 个月后的 mastery**：human 不再写"PRD"，他写**"一个项目的成长档案"**——每个项目积累的不仅是代码，是这个项目的工程偏好（"这个项目我们偏向 SQLite 不要 Postgres"）、领域 lore（"这个 domain 的用户对 sub-100ms 延迟敏感"）、过往决策的回看（"上次我们试了 Redux 后悔了，这次直接 Zustand"）。管道学习这些 lore，新需求来了能引用历史决策。**human 的角色从"写需求"变成"管理一个不断长肌肉的工程团队"**，团队是 agent 但身上有他这个项目独特的味道。

**百万人用了的世界**：这条管道走通后，世界上多了一类新岗位——"PRD 工匠"。他们不是程序员、也不是产品经理（PM 现在产物是 Jira ticket，PRD 工匠的产物是可执行的 PRD）。**软件开发的瓶颈从"代码产出速度"前移到"需求清晰度+工程审美"**。会写 PRD 的人比会写代码的人更值钱。同时，"可信赖 agent 开发"的标准被拉高——不是靠人 review 兜底，而是靠管道自己内嵌多重 quality gate（spec、test、review、deploy）。Open source 社区会涌现出共享的 skills 集合（addy osmani 的 agent-skills 是雏形），就像今天的 npm package 一样。**软件工程从"一群人协作写代码"变成"一群人各自经营自己的 agent 团队，团队之间共享工程纪律"**。

## 2. What's genuinely novel

诚实地说：**这个 idea 的概念层面不新**。Sourcegraph Cody、Cursor、Devin、Sweep、auto-dev、aider、claude-code 自身的 skills——市场上"agent 写代码"的人很多。addy osmani 的 agent-skills 和 superpowers 已经很接近 human 想要的东西。

那真正"没见过"的核心在哪里？**在「PRD-as-input、可信赖-as-output、人不在循环里」这个三角的同时成立**。

- Cursor / aider / claude-code 默认 human-in-the-loop——快但需要持续陪护。
- Devin / auto-dev / Sweep 想做无人化——但做出来的代码"不敢用"，需要人事后大改。
- agent-skills / superpowers 是工程纪律的复用——但还是工具集，没有把"PRD 进去 → 可部署软件出来"组装成完整管道。

005 试图同时拿到所有三角的所有顶点：**输入侧的语义粒度（PRD 而非 ticket）、过程侧的无人化（不需要陪护）、输出侧的可信度（敢部署）**。这三个目标过去都是各做各的，005 的赌注是"它们可以被一套设计同时解决，前提是工程纪律本身被嵌入管道而不是写在 doc 里靠 agent 自觉"。

**真正的创新点不在于做 agent，在于做「工程纪律的硬连线」**——把 spec/test/review/gate 从"agent 应该遵守的原则"变成"管道结构上不允许跳过的强制环节"。这是从软规范到硬约束的转移，类似 Toyota 把"质量是文化"变成"生产线上不合格品物理上下不去"。idea_gamma2 / autodev_pipe 都在尝试这个，但还是用 prompt 来约束，不是用结构来约束。005 如果做对，是"把工程纪律嵌入到 control flow 而非 system prompt"。

## 3. Utility — what users can actually DO

**场景 A：周末把脑子里的小工具变出来**。
某周六下午，human 想做一个"自动整理我桌面截图、按内容打 tag 的 Mac 小工具"。他花 90 分钟写一份 PRD（功能清单、非功能要求、UI 草图），把它丢进管道，去做别的事。傍晚 6 点回来，桌面上有一个能跑的 dmg 包，附带一份 README、一份 test report、一份"我做了哪些超出 PRD 的判断"清单（"你没说要支持 HEIC，我加了；你没说图标要不要点击有动效，我做了静态版本，TODO 标了"）。**他向朋友讲的是**："我写了 90 分钟需求，赚了一个工具。"——而不是"我让 AI 写代码了"。重点不在 AI，在于"产出可信"。

**场景 B：一个有 5 万行代码的现有项目要加新功能**。
human 接手一个开源项目要做"加 SSO 登录"。他写一份 PRD（要支持 Google/GitHub/SAML、要兼容老用户、要有迁移脚本），丢进管道。**这次的关键不是产出代码，是产出过程**——管道先输出一份"现有代码地图"（它读了 codebase 后画的依赖图）、一份"风险评估"（"现有 session 系统在 7 个地方耦合了 password 字段，我会 mock 出来分两阶段重构"）、一份"建议 PRD 改动"（"你说要兼容老用户，但你的 PRD 没说密码迁移路径，请补充——我提供 3 个方案"）。**human 体会到的不是写代码的省力，是「拿到了一个真懂这个 codebase 的资深工程师该有的产物」**。然后才是代码。

**场景 C：作为一个学习放大器**。
human 让管道做一个 idea，但开了"教学模式"——管道每做一个非显然决策，会在 changelog 里多写一段"我这里在 trade-off X 和 Y，我选 X 因为...你以后想换 Y 的话改 Z 处"。半年后 human 回看自己 30 个项目的 changelog，发现自己**在没写过代码的情况下，建立了对工程权衡的真实直觉**——他知道为什么不要全用 ORM、为什么 retry 必须有 jitter、为什么不要在 hot path 里 log。**他向朋友讲的是**："我没学会写代码，但我学会了软件工程的思考方式。"

## 4. Natural extensions

**v0.2 的明显扩展**：管道学会复盘自己——每个项目结束后，自动生成"这个项目我们犯了什么错、应该怎么改进 skills"，并把改进**自动 PR 回 skills 库**。autodev_pipe 已经有 phase-retrospective 雏形，005 把它从手动变成自动反馈环。

**邻近用户群**：
- **小型 startup CTO**——他们已经会写代码但人少，想用管道当"虚拟团队"做边缘项目（admin tool、内部 dashboard）让真人团队专注核心产品。
- **学院派研究员**——他们有 ML 训练能力但不会做工程化（Karpathy 的 autoresearch 用户群），想把研究 prototype 一键变成可部署 demo。
- **产品 PM 转型者**——传统 PM 想绕过工程团队做小工具验证假设。

**邻近产品族**：
- **PRD-quality linter**——独立产品，给 human 写 PRD 时实时反馈"你这条会让管道困惑、需要补 X"。是 005 的前置工具，可以单独卖。
- **Skills marketplace**——共享、付费的工程 skills 包（"金融领域合规 skills 包"、"嵌入式系统 skills 包"）。生态位类似今天的 npm。
- **Agent-team-as-a-service**——把 005 的管道托管化，按项目数/月费收钱。SaaS 化路径。

**最有潜力的扩展**：**让管道反向写 PRD 的辅助工具**。不是替 human 写 PRD（那是 §0(C) 的越界），而是**让 human 用对话+草图迭代出 PRD，然后管道告诉他"这份 PRD 在我看来还有 5 个不确定点"**。这个工具会让 005 的入口体验从"挑战的写作"变成"被引导的清晰化"。

## 5. Natural limits

**005 不应该是这些**：

- **不是「自然语言一句话生成完整 app」**。那是 v0 prompt 工程的浪漫，已经被市场证明产物不可信。005 的核心前提是"PRD 是人写的，且是高质量的"。如果 PRD 烂，005 没法救——它能 surface 烂的地方，不能替你想清楚。**护栏是：管道在 PRD 不够清晰时必须停下来要 clarification，绝不靠 hallucinate 补全**。
- **不是「为大型多人协作团队做的开发系统」**。一旦涉及 5 个以上 human 各自有 agent 团队、互相 review、跨 agent 协作—— governance / 冲突解决 / 责任归属变成主问题，005 不是来解这个的。005 的天花板是**"一个 human + N 个 agent"**的单元生产力。多个这种单元之间怎么协作是另一个 idea（也许是 idea_gamma2 的领地）。
- **不是「面向已有大团队的工具」**。如果一个公司已经有 50 人工程团队、有自己的 CI/CD、有自己的 code style，005 进去会和他们的现有流程冲突。005 的甜蜜带是**"原本没有团队、但需要团队级产出"的个人或两三人创业**。
- **不是「实时协作工具」**。Cursor 的实时协作是另一个产品族——要求 sub-100ms 反应、人和 agent 同屏 ping-pong。005 是异步的——human 提交 PRD 后离开屏幕。这两种协作模式从根上不同，混用会两边都做不好。
- **文化/语言限制**：写 PRD 这件事在英文文档传统里更成熟（PRD/RFC/ADR 都是西方工程文化的产物）。中文场景下"用文档驱动开发"的文化薄弱（更习惯口头沟通+不断改），005 可能在中文母语用户里遇到"懒得写 PRD"的阻力。这是一个需要面对的文化张力。

## 6. Three honest questions about this idea

1. **当 PRD 里有 ambiguity 时，管道应该停下来问，还是合理推断+暴露决策？**——这是核心张力。停下来问会破坏"自动化"承诺，但合理推断+事后说明又可能积累隐性偏差，等 human 察觉时已经晚了 3 层决策。市场上 Devin 类产品在这点上一直没解好。**这个答案决定 005 的体验形状**。

2. **「可信赖输出」的标准是什么？**——passing tests？coverage > 80%？code review pass？human 抽查 10% 没问题？市场上没有共识，每家定义不同。如果 005 的"可信"标准定得低，等于退化为 Devin；定得太高，没有人能用得起（每个项目要花 100 倍成本做证明）。需要找到一个**实操可行+用户敢用**的中间点。

3. **这条管道本身怎么自我演进？**——human 已经做了 4 次尝试（idea_gamma2/vibe-workflow/autodev_pipe/当前 repo），每次都 deprecate 上一次。如果 005 的 v0.1 也只是第 5 次尝试，意义有限。**005 必须解决"管道怎么从每个项目里学到东西并改进自己"——而不是仅靠 human 在每次失败后手工调 prompt**。phase-retrospective 是一个起点，但它需要变成自动+持续的演进环路。否则 005 会重蹈前 4 次的覆辙。
