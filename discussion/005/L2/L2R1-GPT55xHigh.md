## 0. Alternative framings considered

一种读法是“做一个更强的 Claude Code 工作流包”：把已有经验沉淀成更好的命令、角色、检查表和阶段文档。我没有沿着这条路走，因为这会把 005 降格成工具熟练度问题，而 proposal 的真正痛点不是“缺少某几个步骤”，而是操作者不知道什么时候该信任步骤、什么时候该怀疑步骤、什么时候该停下来重问目标。如果后续证明用户只需要提升已有工具的稳定使用体验，我会重新考虑这个 framing。

第二种读法是“做一个个人软件工厂”：PRD 进来，软件出去，中间尽量无人工接触。我没有追这个版本，因为它太容易把“自动化程度最高”理解成“人越少越好”。005 更值得探索的矛盾恰恰是：可靠性通常来自清晰判断、边界控制和证据积累，而不是来自完全消失的人。这里的关键不是把人赶出流程，而是让人的参与从频繁救火变成少量、高杠杆的裁决。

第三种读法是“给非工程背景创始人的外包替代品”：用户写需求，agent 团队像廉价工程团队一样交付。我没有采用，因为 proposal 里的用户并不是普通不会写需求的人。他有 CS/ML 背景，能构造 PRD，也愿意做反思，真正缺的是工业软件开发中的肌肉记忆和质量直觉。因此 005 不是“代替工程师”，而是把工程组织的判断习惯压缩成一个可被单人持续使用的协作制度。

第四种读法是“把四个旧项目经验合并成最终版流程”。我也没有选它，因为“最终版流程”本身可能是陷阱。过去一个月连续出现 idea_gamma2、vibe-workflow、autodev_pipe、当前 repo，说明问题不只是缺少一个更完整的模板，而是缺少一种能吸收旧尝试、保留有效结论、淘汰错误假设、同时不让整个系统推倒重来的演化方式。005 最有价值的形态应该是抗重做的，而不是第五个大一统版本。

## 1. The idea, unpacked

我最终把 005 理解为“面向强需求表达者、弱工程交付直觉者的自动化开发治理框架”。它的用户不是所有程序员，也不是完全不懂技术的人，而是一个很特殊的人群：能把产品意图、约束、验收标准写得相对清楚，能理解模型输出的高层逻辑，但缺少多年工程实践带来的直觉。例如，一个 ML 研究者想把研究工具变成可长期使用的软件；一个独立创业者能写出完整 PRD，却不知道哪些需求会在交付中失控；一个个人效率系统爱好者能设计复杂流程，却不确定什么样的质量证据才算足够。

在拥有 005 之前，这类用户每天的状态是高认知负担但低确定性。他们可以启动很多 agent，可以写越来越长的说明，可以让模型生成计划、复盘、技能、角色，但心里始终有一个问题：我现在看到的是专业工程推进，还是一套看起来专业的文本仪式？一旦项目变大，用户会开始反复重建流程，因为每次失败都像是在提醒他“这套方法不够专业”。于是旧流程被归档，新流程被命名，新一轮希望开始，但真正的判断资产没有稳定累积。

005 改变的不是“agent 帮我写更多东西”，而是“我终于知道每个阶段该相信什么”。PRD 不再只是给 agent 的输入，而是进入一套从意图、范围、任务、验证、评审到交付信心的连续语言。用户的 aha moment 不是看到代码生成，而是在前 30 秒看到系统把他的需求翻译成一组清晰的承诺：哪些是产品决定，哪些只是待澄清问题，哪些需要人类裁决，哪些可以交给自动化推进，哪些证据会让系统说“可以继续”。这种体验会让用户第一次感觉，自己不是在哄模型做事，而是在操作一个有纪律的交付机构。

六个月后的熟练用户会变得很不一样。他不需要每天亲自扮演资深工程经理、质量把关人、范围守门人和发布负责人；他只需要在少数节点做高质量选择。他会积累一套自己的项目史：哪些 PRD 写法会导致漂移，哪些边界表述能显著减少返工，哪些“看起来聪明”的自动化其实制造幻觉，哪些人工裁决最值得保留。005 因此不是静态 pipeline，而是一种把工程经验外置化的长期学习伴侣。它越用越像用户自己的交付风格，而不是某个社区最佳实践的复制品。

如果一百万人使用类似 005 的东西，变化不只是更多人能“vibe coding”。更大的变化是，软件创造的门槛从“你是否有工程团队”转向“你是否能清楚表达目标，并愿意接受纪律化的澄清”。大量研究者、设计师、运营者、小企业主会把软件视为可迭代的工作资产，而不是一次性外包项目。与此同时，好的需求表达、好的范围取舍、好的验收语言会变成个人生产力的一部分。这也意味着 005 的文化目标并非让人无脑自动化，而是让非工程操作者获得工程化表达的尊严和控制感。

## 2. What's genuinely novel

005 的概念材料并不全新：agent skills、专业流程、阶段检查、评审约束、自动化开发，这些方向都已经有清晰先例。真正新鲜的部分在组合重心：它不是为专业工程团队提速，也不是为新手隐藏复杂性，而是为“能写需求但缺少交付直觉”的单人操作者提供一套可演化的信任系统。

它相对 addy osmani 式工程纪律迁移的增量，是把“通用专业习惯”改造成“个人长期校准机制”。它相对 superpowers 式能力扩展的增量，是不把重点放在 agent 会做多少事，而是放在什么时候应该让 agent 做、什么时候必须暴露不确定性、什么时候应要求人类做少量决策。这里的 novelty 不是某个模块，而是将自动化和可靠性的冲突显性化：最高自动化不是最少人工，而是最少低价值人工、最多可审计信心。

## 3. Utility — what can users actually DO with this?

场景一：一个 ML 博士想把实验室里反复使用的研究脚本变成小团队可用的内部工具。他能写清楚“谁用、用来做什么、成功是什么样”，但不确定开发过程该如何分阶段。005 帮他把 PRD 变成一组可被反复确认的交付承诺，并在每个关键点提醒他只回答产品层面的选择。他完成的是从研究想法到可用软件的转化，消失的是“我是不是漏了专业流程”的持续焦虑。他会对朋友说：它不是替我思考，而是让我只在真正需要我判断的地方思考。

场景二：一个独立创业者已经让 agent 做过几个 demo，但每次 demo 到正式产品之间都会散掉。她使用 005 后，不再把 PRD 直接丢给模型期待奇迹，而是先得到一张关于范围、风险、验收、暂停条件的清晰地图。她可以决定哪些体验必须保留，哪些先不碰，哪些问题还不值得进入开发。她完成的是从“看起来能跑”到“我知道为什么它可以继续”的心智转变。她会说：以前我是在催模型交东西，现在我是在管理一个会留下证据的交付过程。

场景三：一个没有工业开发经历的个人系统构建者，过去一个月已经迭代了四套 agentic coding 方法，每套都曾经看起来像答案。005 对他的直接用途，是停止“废弃后重做”的循环：每次项目推进后，系统不只是生成新流程，而是沉淀哪些判断有效、哪些假设错了、哪些人工介入是必要的。下一次 PRD 进入时，他面对的不是空白新开始，而是一套带有历史记忆的个人工程制度。他会说：这次失败没有白费，它变成了下一次更少返工的依据。

## 4. Natural extensions

第一个自然延展是“个人工程风格画像”。当用户运行多个项目后，005 可以逐渐总结他常见的 PRD 盲点、范围偏好、验收松紧、决策延迟模式，让下一次探索更贴近这个人的真实习惯。

第二个延展是“项目类型手册族”。研究工具、内容产品、内部运营工具、面向消费者的小应用，虽然都可从 PRD 出发，但它们需要的取舍语言不同。005 若 v0.1 成立，用户自然会要求它识别不同项目气质，并给出不同的澄清重点。

第三个延展是“多人轻协作”。单人操作者可能会邀请一个设计朋友、一个领域专家、一个真实用户参与少数节点。005 可以成为他们之间的共同语言，让参与者只在自己有判断力的地方发声，而不被完整开发流程吞没。

## 5. Natural limits

005 不应该承诺“任何 PRD 自动变成可靠软件”。有些 PRD 本身是愿望清单，有些产品价值未经验证，有些需求互相冲突；L2 层面只能说，这类输入需要被暴露和澄清，不能被自动化包装成确定性。

它也不适合替代真正的组织责任。大公司、强监管领域、多人长期维护系统都有复杂的人际、合规和责任结构，005 可以提供语言和纪律，但不应伪装成完整组织。

最大的 scope creep 陷阱，是把它扩成“万能 AI 工程公司”。一旦 005 同时追求项目管理、团队协作、代码生成、知识库、教育课程、外包平台，它的核心会变糊。它最该守住的是：帮助一个会表达需求的人，以最少但最高价值的人类裁决，获得对自动化开发过程的可解释信心。

## 6. Three honest questions about this idea

1. 对目标用户来说，“可靠”最真实的含义是什么：少 bug、少返工、范围不漂移、可持续维护，还是心理上敢继续推进？
2. 过去四次尝试中，真正导致不满意的是产物质量、流程不可复用、人工介入太多，还是缺少一个能解释失败原因的反馈循环？
3. 相比现有 agent-skills / superpowers 等实践，真实用户最愿意为哪一种增量买单：更少操作、更强信心、更好决策，还是失败后更容易复盘？
