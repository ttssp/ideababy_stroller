## §7 Known gotchas

1. **try_claim_expand 单连接序列化**:依赖 `write_connection()` writer_lock + 同 conn 多 SQL + 单 commit。SQLite 单连接事务 atomic · cursor.rowcount==0 时 UPDATE 0 行无副作用 · 不需 rollback。但若未来 INSERT 抛异常,需补 try/except + rollback(A2 backlog · T040 真 LLM 时加)

2. **`{term:path}` converter 副作用**:
   - `/concept/explain/` (空 term)→ FastAPI 404(无 routes 匹配)· 设计 intent
   - `/concept/explain/a/b/c` (多 slash) → term="a/b/c" 全捕 · repo.get_by_term("a/b/c") 必返 None → 404 · 真实数据库 term 不太可能含多 slash · backlog A4

3. **mock LLM 不真模拟真 LLM 行为**:`score_expand` 返 const f-string · 不真"展开新解释"语义。T040 升真 LLM 时:
   - 接 Anthropic LLMClient.call(model=Haiku · cost cap $40/月)
   - 加 try/except + fallback definition_text(LLM 失败不破 short-circuit gate)
   - 加 ExplainResult.explanation 真 LLM 输出(可能长 + Telegram 4096 char 限制需切)

4. **双路径返 explanation 不同**:
   - Web JSON:explanation 字段直接放 mock LLM 输出(短路时 = definition_text)
   - Telegram text:`🔍 短路` 或 `📖 新解释` marker + explanation
   - 真实前端调用:JSON 路径前端自渲 / Telegram 直显
   - test 用 marker 模糊匹配("短路" / "已解释" / "30 天" any-of)· 模板未来改字符串时 test 需同步

5. **`update_last_explained_at` 在 race lost path 未调用**:fast short-circuit 不更新 last_explained_at(idempotent 短路) · race lost path 也不更新(claim 失败 = 别人抢先已 update 过)· 这是 design intent · 双信号(last_explained_at + log)只在真 expand 路径更新

6. **件 3.1 阶段 2c 波 3 闭环**:T010(`db09f00`)+ T011(`e0c2c48`)+ T012(`36eb012`)= v0.2 Phase 2a 全 ship · T040 入度全就绪 · 可起波 4(T013 / T020 / T024 opus 重件 OR T040 integration test 聚合)
