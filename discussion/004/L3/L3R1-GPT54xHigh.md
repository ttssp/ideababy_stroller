## 0. How I read the intake

我把这次 intake 读成一个很明确的绑定题：v0.1 不是去证明“阈值多准、私有模型多强、咨询师解析多深”，而是要在 45-180 小时里把一条每周可运行的 pipeline + shell 跑通，让 Telegram bot 和本地 Web UI 共同承载“导入咨询师 PDF/文本 → 映射到关注股与持仓 → 显式展示咨询师/私有模型/agent 三路冲突 → 给出动作或不动作建议并白话解释 → 留痕复盘”。我会严格尊重 🧭 原则：动手阈值、私有模型、解析深度都只做接口 + 占位实现，不拿它们做候选差异。候选只在 shell 宽度、平台主次、记录深度上分化，同时都 honor 每周 1 次 + event、显式冲突报告、Telegram + 本地 Web UI、单人自用、PDF/文本先行、以及 L2R3 的 8 条红线。

## 1. Candidate A · “周报驾驶舱”

### v0.1 in one paragraph
这是一个 **Telegram 主导、Web 补充** 的周度决策壳。每周你把咨询师 PDF/文本导入后，系统生成一份周报：本周主张、映射到你关注的 30-50 只股票、三路冲突报告、1-3 个建议动作或“不动”建议、以及每条建议的白话解释。Web UI 只承担图表和历史查看，不承担重度操作；event 触发时只发简短提醒卡，不扩展成高频盯盘机。

### User persona
同一个 human 的“碎片时间执行版”：平日主要在手机上消费信息，周末愿意花 20-30 分钟完成一次集中判断，金融仍偏初级，需要 agent 先翻译再建议。

### Core user stories
- 作为订阅咨询师的人，我能把本周内容导入并快速看到它跟我关注股的对应关系。
- 作为容易被多路观点拉扯的人，我能看到咨询师、私有模型、agent 三路分歧及根因。
- 作为稳健型投资者，我能收到“该动”与“不要动”的对等建议，而不是被持续催交易。
- 作为在手机上完成大部分决策的人，我能在 Telegram 里完成查看、标记 watch、记录最终决定。

### Scope IN
- 每周 1 次周报生成
- event 简短提醒卡
- 显式冲突报告
- 关注股/持仓映射
- 动作或不动作建议 + 白话解释
- 简单决策留痕

### Scope OUT
- 深度决策档案与长篇复盘
- 全市场事件日历
- 视频/音频实解析
- 自动抓取微信内容
- 每日晨报或盘中频繁提醒

### Success looks like
- 连续 4 周都能从导入咨询师内容走到清晰的“做/不做/watch”结果
- 单次周报处理时间压到 30 分钟以内
- 每周至少出现 1 次“明确选择不动且理由可复述”
- human 不再需要手工对照咨询师观点和自选股

### Honest time estimate under 45-180 hour budget
- 约 4-5 周，80-120 小时
- Confidence: 高。因为 shell 窄，重点放在“周报能闭环”

### UX principles
- 低摩擦 > 全面记录
- Telegram 处理效率 > Web 完整性
- 先给清晰结论，再展开白话解释

### Biggest risk
最大的风险是它可能太“轻”，能证明周报闭环成立，但未必足够证明“机制成立”。如果 4-6 周后你只有建议列表，没有足够的历史上下文和复盘证据，你会知道系统好用，但不一定知道它是否真的在减少冲动错误。

## 2. Candidate B · “决策档案台”

### v0.1 in one paragraph
这是一个 **Web 主导、Telegram 只负责提醒和入口** 的 calibration-first 壳。周报、event 卡、冲突报告都仍然存在，但 v0.1 的核心不是“本周给你几条动作”，而是把每次动作/不动作背后的环境、理由、分歧和事后结果都沉淀成决策档案。Web UI 是主场：你在里面看错位矩阵、分歧记录、历史判断；Telegram 只把你带回桌面进行正式判断。

### User persona
同一个 human 的“学习进化版”：愿意每周多花 30-45 分钟复盘，希望 3 个月后能明确看见自己到底在哪些情境最容易犯错。

### Core user stories
- 作为想边做边学的人，我能看到每次建议背后的白话解释与关键依据。
- 作为会受咨询师权威影响的人，我能回看自己到底是按了谁的意见，以及结果如何。
- 作为想减少冲动的人，我能把“不动”也作为正式决策记录下来。
- 作为要验证机制的人，我能在几周后回看一组结构化档案，而不只是零散通知。

### Scope IN
- 每周周报
- event 卡
- 显式冲突报告
- 决策档案与历史回看
- 错位矩阵/持仓分布等研究型视图
- 学习性解释与术语回顾

### Scope OUT
- 自动化抓取和多模态深解析
- 高频提醒
- 复杂绩效归因
- 回测环境
- 面向他人的 polished 产品体验

### Success looks like
- 4-6 周内沉淀出 8-12 条完整决策档案
- human 能清楚说出自己 1-2 种常见错误模式
- 至少有一组“咨询师/模型/agent 分歧”被看清并留下后续结果
- 稳定期维护时间有希望压到每周 3 小时以内

### Honest time estimate under 45-180 hour budget
- 约 5-6 周，110-160 小时
- Confidence: 中。原因是记录深度容易膨胀

### UX principles
- 可复盘性 > 即时爽感
- 把“不动”当一等公民
- 解释质量 > 推送频率

### Biggest risk
最大的风险是“自用工具最怕 upkeep”。如果档案字段太多、每次判断都像写作业，这个候选会在价值还没显现前先把你拖累。它最有机会证明机制，但也最容易因为记录负担过大而死。

## 3. Candidate C · “事件值班台”

### v0.1 in one paragraph
这是一个 **周报 + event 双核心** 的平衡壳。它不像 A 那样把一切压缩进周报，也不像 B 那样把重心放在完整档案，而是把“重大事件前后的决策准备”做成 v0.1 的第一性体验：周报给出本周基线，FOMC/财报/重大异动来时生成事件卡，明确受影响持仓、三路分歧、建议动作或等待、以及事件后应如何记录结果。Web 与 Telegram 都重要，但都服务于“事件前先想清楚”。

### User persona
同一个 human 的“关键节点防冲动版”：平时不想被打扰，但一到重大事件前就最容易犹豫或乱动，需要 agent 在关键时点把复杂信息压成可执行判断。

### Core user stories
- 作为会在重大事件前紧张的人，我能提前看到哪些持仓会受影响以及为什么。
- 作为不想错过窗口但也不想乱动的人，我能拿到明确的“现在不动/等确认/可小幅动作”建议。
- 作为单人自用者，我能把事件前判断和事件后结果放在同一条时间线上看。
- 作为需要保持节制的人，我只在周报和真正重要的 event 时被打扰。

### Scope IN
- 每周周报
- 事件准备卡与事件后记录
- 显式冲突报告
- 受影响持仓/关注股清单
- 简化时间线与决策留痕

### Scope OUT
- 每日盯盘
- 覆盖所有宏观与个股事件
- 全量复盘系统
- 视频/音频实解析
- 自动执行建议

### Success looks like
- 4-6 周内至少覆盖 2-4 次真实 event 决策
- 重大事件来时不再临场拼接信息
- 通知频率仍被控制在“每周 + event”边界内
- human 能区分“这次该动”与“这次只是焦虑”

### Honest time estimate under 45-180 hour budget
- 约 5-6 周，100-150 小时
- Confidence: 中。取决于事件壳做多厚

### UX principles
- 关键时点清晰度 > 平时功能密度
- 节制提醒 > 覆盖一切
- 先保护不犯大错，再追求多做动作

### Biggest risk
最大的风险是样本密度不足。若 build 窗口内真正值得记录的 event 不多，这个壳可能显得很对，但证据不够厚；你会觉得它有前途，却很难在 v0.1 阶段判断它是不是最值得先做的主壳。

## 4. Options for ❓ items from intake

### 1. “冲突报告”形态
- 方案 1（推荐）：**混合式**。先给一句结论，再给三路矩阵，最后用一小段白话解释分歧根因。
- 方案 2：**表格式**。最快扫读，最适合 A/C，但学习感较弱。
- 方案 3：**叙事式**。最适合 B，教育价值高，但会拖慢单次判断。

### 2. v0.1 “机制成立”的可观察证据
- 方案 1：**流程证据**。连续数周都能稳定完成“导入 → 判断 → 留痕”，且单次决策耗时下降。
- 方案 2：**校准证据**。开始看得见自己在哪些情况下更该按咨询师、按 agent、或选择不动。
- 方案 3：**学习证据**。几周后你能复述最近频繁出现的 5-8 个概念，并解释至少 2 次“不动”的理由。

### 3. 咨询师 pipeline v0.1 scope
- 方案 1（推荐）：**只收 PDF/文本，人工触发导入**。最符合绑定原则。
- 方案 2：**PDF/文本 + 图片 OCR/粘贴入口**。如果咨询师大量发截图，可多给一点便利。
- 方案 3：**PDF/文本完整进入，音频/视频只做占位队列**。即先能登记、分类、等待后续迭代，不在 v0.1 真解析。

## 5. Red lines

我明确 honor L2R3 的 8 条红线，并再补两条更贴近 L3 scope 的红线：

1. **不允许静默偏向某一路**：任何建议都必须显式展示咨询师 / 私有模型 / agent 三条泳道，不能偷偷给默认优先级。
2. **不允许在证据薄弱时硬给动作**：`等待更多确认` 与 `继续不动` 必须是正式、可推荐、可记录的输出，而不是失败状态。

## 6. Questions that need “real user interviews”

这里翻译成“human 在开工前该先问自己什么”：

1. 拿最近 3 期咨询师内容，分别模拟 A/B/C 的使用方式各 20-30 分钟，哪一种最像你真的会持续使用的节奏？
2. 你到底更想在手机里结束 80% 的决策，还是只想在 Telegram 收到入口、真正判断必须回到桌面？
3. 你能不能接受为每次动作或不动作写 2-3 句理由？如果连续试 5 次就嫌重，B 可能不对。
4. 回看最近两个月的重大事件，你真正后悔的是“错过动作”，还是“在焦虑里乱动”？这个答案会直接决定你该选 A、B 还是 C。
