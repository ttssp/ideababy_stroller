## §1 Rationale(执行摘要)

本次 dogfood 起于 operator 配新手机代理 token 后的两个运维诉求:核对图文/问答增量是否 updated、补齐没下完的回放。执行中链路全程跑通并暴露/澄清了若干事实:

**回放补漏(已完成)**:回放全集 355 个,缺口 43 个公开直播音频原标 `formatMediaUrl 未返签名(code=1020002)`。初判疑为「公开直播 f0.mp4 源形态不可签」,经实跑换签名验证**推翻**——根因是当时 token 过期,token 新鲜即可签。43 个全是 1h+ 完整长录播、源不支持 Range,ffmpeg `-vn` 不省流量(~12min/条),单 token 窗口下不完。operator 决策「串行+跨 token 窗口多批续」,经 3 个 token 窗口接力**全部下完,全集音频 100% 落盘**。

**增量链路验证 work**:token 新鲜时 xeno-fetch 按 30min 节奏抓取,实证当天发布的图文(如 Joe 交易日笔记、亚盘技术面观测)当天入库 + DeepSeek 结构化提取(key_points + ticker 识别 + source_ref 逐字回链 C6 校验);token 过期则静默跳过、刷新后批量补回。checkpoint「captured 但 DB 无」盲区经守护后续轮自愈归零(DB form=replay 290→312)。

**借机系统盘点(本 hand-back 的真正触发)**:三品类共 **8913 条**入库(图文 1150 / 问答 7451 / 回放 312),图文/问答结构化覆盖率 99.97%(各仅 1 条因 DeepSeek 稳定返回坏 JSON 缺 summary)。盘点中 operator 提出关键质疑:**「结构化的数据后续怎么用?下游用途/目标如果没 ready,前期结构化会不会纸上谈兵?」**

为系统回答,用 6 个独立 subagent 并行从真实样本(图文 100 + 问答 120)**自底向上归纳**结构化字典,merge 去重后产 `docs/STRUCTURE-DICTIONARY-draft.md`,划出三层(按依赖下游目标程度):第1层无悔(修 bug/填信息黑洞/显式可靠)、第2层赌用途(高价值但锁在未确认下游)、第3层结构关系(架构对但全量重构是赌博)。再经一轮红队 adversarial 评审收敛「无悔层」边界——红队推翻了初版把 `conviction`/`stance_direction` 等当无悔的划分,指出它们价值全锁在「准确率追踪」这一未确认下游、且在反讽语料提取不可靠。

operator 确认本语料**永远只出自 trader韭团队一家**(不扩展其他作者)→ 字典可针对性过拟合该团队文体(右侧/左侧方言、四桶仓位、黑话谐音、期权打法),作者特异性是 feature 不是 bug。

## §2 触发原因 + 决议链(按 frontmatter `tags` 逐条说明)

本包性质 = **PRD 缺口上报 + dogfood 回流**,非 spec/PRD 矛盾,非重大架构转向。三条触发:

**① 下游用途/消费目标从未定义(根因 · PRD 缺口)**。008-pB PRD/spec 通篇定义的是「采集稳定性」(gate 判准 = build-runtime 可自证工程事实;合规/用途由 operator 负责不进 gate),**没有任何一条定义「结构化数据被某个下游消费并产生价值」**。后果:结构化字典第2层(方向/确定性/情绪/仓位/财报预期差…)、第3层(层级树/关系/跨文档/细粒度)**全部悬空**——无法判断该不该提取、该提哪些。继续在 build runtime 堆字段 = operator 担心的「纸上谈兵」(为没确定的用途过度设计)。这正是「PRD 约束/目标不清」该切回 IDS 的信号(per CLAUDE.md 决策矩阵)。

**② operator 已倾向「机器可信信号/跟单」,但提取可靠性未实测(可行性悬而未决)**。operator 表态:下游想做「自己回看检索 + 提炼投资信号/跟单」,且要「机器可信信号」(能半自动跟单/报警)。但红队评审点破其**真瓶颈不是 schema 设计,是提取可靠性**:trader韭团队满嘴反讽(「这票真是好啊」可能看空)+ 黑话(达子/卤馒头/谷子…),LLM 判 `stance_direction`(看多空)/`conviction`(确定性)/`action_type`(加减仓)在此语料**中等易错**——而这恰是跟单依赖的核心字段。信号错标 = 真金白银亏损,比无信号更危险。**关键空白**:LLM 在本语料提取这些信号的准确率**从无实测数字**,红队只是定性推测。在拿到真实准确率前,「机器可信信号/跟单」**可行性无法判定**。

**③ 数据缺陷两条(随包回流,攒批 forge,非本包当场改)**:
- **KG-24**(框架级 · high):增量守护 `_pending_replays_fn` 只认 manifest `status=downloaded`、不认 `skipped`。多 token 窗口重跑下载时,已下产物被断点续标 `skipped` → 守护视而不见、漏 ASR(本次 31/43 被漏,守护只能补 12)。两处对 manifest 状态语义理解不一致。受益面 = 所有「批量下载(含断点续)→ 守护增量消费」的链路。
- **KG-25**(框架级 · medium):DeepSeek 结构化提取无坏-JSON 容错。LLM 对特定正文**稳定**返回非法 JSON(未转义控制字符,连试 3 次同位置失败),`json.loads` 一次失败即抛、被守护静默吞 → 该条永久缺 summary、operator 无从知晓。受益面 = 所有用 LLM 做结构化提取的项目。

两条已记 XenoDev `dogfood-backlog.md`(KG-24/KG-25,含真实复现 + forge 候选解法),按 dogfood 铁律不在 XenoDev 当场改框架,随本包回流统一 forge 批审。

## §3 Suggested IDS actions

给 IDS forge 的 X 标的(建议 `/expert-forge 008` 决议):

- [ ] **PRD 补「下游用途/消费目标」定义**(根因)。候选用途(operator 已倾向后两项加粗):自己回看/检索(轻量)、**提炼投资信号/跟单(机器可信)**、对外产品(观点聚合/情绪指数/板块轮动图谱)。用途定了,结构化字典第2/3层才有取舍依据。

- [ ] **若定「机器可信信号/跟单」→ 先决议做「信号提取可靠性 spike」**(本包核心建议)。理由:跟单真瓶颈是提取可靠性非 schema,而 LLM 在本反讽+黑话语料的提取准确率**从无实测**。spike 内容:小样本(~100-200 条)让 LLM 提 `stance_direction`/`conviction`/`action_type` → operator 或对照原文人工核准确率 → 拿真数字再判:① 跟单是否可行 ② 是否需置信度门槛(只保留 high-confidence 信号)③ 反讽/黑话误判率上界。**这是 XenoDev 能直接产的硬证据,但需 forge 先把它列为 PRD 工作项**(避免 build runtime 静默决定产品方向 = V4 失败模式)。

- [ ] **据用途从结构化字典三层挑工作项**(`docs/STRUCTURE-DICTIONARY-draft.md`,随本包参考):
