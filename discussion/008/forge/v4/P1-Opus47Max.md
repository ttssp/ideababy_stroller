## 1. 现状摘要(按 Y 视角组织)

### 视角 A · 产品价值/用途成立性

**关键事实:PRD 不是「完全没定下游」,而是把下游价值显式划给了 004。** PRD Problem/Context(line 61-62)钉死:「008 = …作为 **004 投资决策智能体的上游数据采集模块**…只负责把顾问发的东西完整、及时、省心地收进来并可消费,**不产生投资决策**(那是 004)」。US6(v0.1)= 「把收下的内容(轻量格式)**喂给 004** / So that 004 能引用顾问观点做决策」。给 004 的出口(line 93)= 「**轻量输出包**(粗约定:发布时间、形态、原始内容/转录、关键点、涉及标的、采集状态)」。验收 O5 = 「004 能稳定拿到边界清楚的轻量包,且**不误认 008 在给建议**」。

**所以「下游用途未定义」的精确位置 = 008 与 004 之间的接缝。** 008 侧 PRD 完整(采集稳定性 + 轻量粗约定出口);**004 侧的消费需求从未被写**。字典草稿的「赌用途层」字段(stance_direction / conviction / action_type / positions / vs_consensus…)正好坐落在这条接缝的无人区:对 008 的「轻量粗约定」过深,而 004 要什么没人定义。operator 新倾向的「机器可信信号/跟单」**本质是 004 的职责回流到 008 采集层**。

**这里有一条 hard red-line 张力被低估了**:PRD Scope OUT(line 152)+ C9(line 227)明令 008「❌ 产生投资建议/买卖信号/仓位建议(红线——是 004 的职责)」「任何标签/要点可回到顾问原话」。把 stance/conviction/action 作为「**顾问原话的可溯源结构化**」存(谁说的、原话在哪)= C9 内、不踩红线;但若包装成「**机器可信跟单信号**」(系统替 operator 判方向)= 已越界成 004 的活,且与 008 红线直接冲突。同一批字段,**定性不同,合规与职责归属就不同**。

### 视角 B · 提取可靠性/可行性

字典草稿自己已经把可靠性问题摊开,且红队做过一轮收敛:
- `stance_direction`「⚠ 反讽语料中等易错,做须带 `_confidence`」(line 35/67);
- `conviction`「⚠ 语气判断在反讽语料不可靠」,自报「准确率追踪命门」(line 36/66);
- `author_tone` 的 `humorous_sarcastic`「最高频也最易错」(line 37/68)。

红队的核心判断(line 13/79):**「信息价值高」≠「无悔」——价值高但抵押在未确认下游、且反讽语料提不可靠的,踢出无悔层**。这正好命中 operator 的跟单倾向:跟单依赖的恰恰是被踢出的那三个字段。

**关键空白(我判断这是 v4 最 load-bearing 的事实)**:全语料对这三个字段的 LLM 提取准确率**从无任何实测数字**——红队是定性推测,字典草稿是定性标注。「中等易错」是个没有分母的形容词。在跟单场景里,**错标的代价不对称**(假信号→真金亏损 > 漏信号→错过),所以「中等」这个模糊档位根本不足以支撑 go/no-go。

### 视角 C · 数据建模/字典设计

字典质量很高:自底向上从真实样本归纳(非拍脑袋)、红队批判过、三层按「依赖下游程度」分得干净、执行落在「方案 B 轻量可逆」(每条只多 1-2 字段,第 2/3 层冻结)。第 1 层无悔(asker bug / entity_type 板块黑洞 / tech_side 三值 / 黑话词典)论证扎实:`asker` 是真数据正确性 bug(records.author 存的是答主、提问人埋正文),`entity_type` 补的是 73-85% key_point 无 ticker 的板块层黑洞(60% 主语丢失)。这些与用途无关,**确实可以不等 forge 先落地**。

可商榷点:红队把 `asker` 自己又降为「搭便车字段」(依附 entity_type 重提),而真正硬的是 `entity_type`。我倾向同意——`asker` 是 bug 但低频价值,`entity_type` 是不补则永久丢的黑洞,二者「无悔」的紧迫度不同。

### 视角 D · 工程纪律

KG-24/KG-25 都是诚实、可复现、根因清晰的框架级缺陷:
- **KG-24(high)**:`_pending_replays_fn` 只认 `downloaded` 漏 `skipped`,与正牌入口 `run_asr_from_manifest`(认两者)**状态语义不一致** → 多 token 窗口断点续下载后,31/43 已下产物被守护视而不见、永久漏 ASR、无告警。根因不是「漏写一个枚举值」,是 **manifest 作为多写者共享 append-log 缺统一状态契约**。
- **KG-25(medium)**:`make_deepseek_text_summarize` 直接 `json.loads` 无容错,LLM 对特定正文**确定性**返回坏 JSON(连试 3 次同位置失败)→ 静默吞 → 每轮重选每轮失败、summary 永久缺、operator 无从知晓。根因是「调 LLM→json.loads」链路**缺解析鲁棒性层**,面广(三品类摘要都走 make_deepseek_*)。

两条都已记 backlog,operator 已走正牌入口手工补数(KG-24 等 GPU 空闲 `batch --asr-from-manifest`;KG-25 这 2 条暂缺)。**作为框架级缺陷,候选解法的取舍是真问题**(KG-24:一行修 vs 抽状态契约 vs 合并语义;KG-25:清洗 vs 重试 vs 鲁棒解析 vs 降级 vs 上报)。
