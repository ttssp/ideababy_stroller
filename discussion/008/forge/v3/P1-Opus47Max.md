## §1 · 现状摘要(按 Y 视角组织)

### §1A · 架构设计视角 —— 当前「半自动被动监听」架构的物理隔离是怎么成立的

现状采集 = `src/capture/addon.py` 旁路挂在 mitmdump 上,**operator 手动在 App 里浏览**触发 HTTPS 流量,
addon **被动监听**并按 C5 规则落盘。这个架构有一个**未被任何协议条款显式命名、但实际承载了凭据隔离的
关键性质**:**agent 物理上不经手登录态**。登录态(cookie/token)只存在于 operator 的真实客户端会话里,
addon 看到的是**已经发生的**流量,agent 看到的是 addon **已经落盘的**结果。三层(operator 会话 → addon
被动落盘 → agent 读落盘)之间,凭据从不向 agent 方向流动。这不是靠一条 deny-list 挡出来的,而是**架构
形态本身**使 agent 没有持 key 发请求的能力 —— 我称之为「被动性即隔离」。

自动翻页**根本上改变这个性质**:翻页 = **主动发请求**抓列表的下一页。主动发请求**必须持有效登录态**
(K 与 backlog 都确认:列表接口要带 operator 登录态)。于是出现一个当前架构里**不存在的主体**:一个
「持 operator 登录态、主动发请求」的东西。议题的全部架构重量都压在这个新主体上 —— 它放哪、谁信任它、
它和 agent 的边界在哪。backlog 的洞察很准:**「删 C5 也解不了」**,因为拦路虎不是「能不能爬」的合规问题,
是「持 key 主体如何与 agent context 物理隔离」的架构问题。

### §1B · 安全/凭据边界视角 —— 「凭据隔离」到底成文约束了什么(written vs 外推)

读 §2 后我必须诚实指出一个 load-bearing 的事实(K 第 3 点其实已隐含问到):**K 称作「三件硬约束①·
登录态绝不进 agent context · non-overridable」的东西,在 §1+§2 原文里没有字面成文条款。** §2件1
（Production credential 物理隔离)成文 scope 是 `.env.production` / `secrets/production/*` / `prod://`
连接串 / KMS 注入的运行时凭据 —— 它针对的是**部署/生产密钥**,防的是「Cursor 9 秒删库」那类把 prod
凭据喂进 agent 导致不可逆破坏的事故。**它字面没说「顾问平台采集会话的登录态 cookie」**。XenoDev 那个
`scan-credentials.sh`(件1 落地)扫的也是 `.env.production` **文件模式**,不是运行时「采集进程持 cookie」。

这意味着:K 里那条「登录态绝不进 agent context」是 §2件1 **精神**在**采集域**的**合理外推**,但**不是
被一条现成条款挡住的**。这恰恰是 K 第 3 点该解的题 —— 而我的初判是:**这道 written-constraint 的空白
不是漏洞,是机会**。因为 §2件1 的**原则**(凭据不进 agent context、防不可逆/越权)外推到采集会话登录态
是**完全站得住的**;v3 真正要做的不是「论证能不能放宽硬约束」(K 已 binding 说不放宽),而是**把这条
目前靠架构隐式成立、靠精神外推的约束,显式写成协议条款**,让它在「自动翻页」这个新场景下有成文依据。
换句话说:§1/§2 **大概率需要改**(K 第 3 点答案倾向 yes),但改法不是「放宽」,是「补一条采集会话凭据
隔离子条款」把隐式约束显式化。

### §1C · 「IN 放宽」与「OUT 渠道中性」是两件正交的事(防与 v2 混淆)

读 PRD C5 + v2 stage 后必须钉死一个边界,否则双专家会把 v2 已决的事当本轮议题:
- **v2 渠道中性 = OUT 侧**:管的是采集**之后**,原内容能不能流出本地到会被同步/传播的衍生渠道(vault /
  iCloud / git)。结论=原文永久 OUT,vault 只放指针+摘要。
- **v3 自动翻页 = IN 侧**:管的是采集**本身**,怎么把 addon 被动监听够不到的 2026 列表页**抓进来**。
- 两者**正交**:v2 verdict 对 IN 侧采集架构**只字未提**,凭据隔离也不在 v2 审议范围。**所以这是 fresh
  forge 不是重审 v2。** 但有一个**交叉点**值得 R2 留意:自动翻页若抓到更多原内容,这些原内容进入 OUT
  侧时仍受 v2 渠道中性管辖 —— IN 侧放宽**不松动** OUT 侧任何约束。

## §2 · First-take 评分(按 Y 维度:keep / refactor / cut / new)

| 项 | 评分 | 理由(基于 X 实读) |
|---|---|---|
| **当前半自动被动监听采集架构** | **keep** | 「被动性即隔离」是当前凭据安全的地基,任何方案都不该拆掉它作为**默认路径**。即使引入自动翻页,被动监听对已可达内容仍应保留。 |
| **候选① 半自动 + agent 侦察兵**(agent 分析 addon 落盘指方向,operator 仍手动翻) | **new(强候选 · 我的初步首选)** | 安全边界**零改动**(agent 仍不经手 key),却把「盲滑」变「定向滑」实打实降 operator 成本。它**不解决**「列表接口本身截断到 2025-07-21」这个**接口层**问题——侦察兵能告诉 operator「2026 在哪个 detail 入口」,但若列表 API 物理上不返回 2026,侦察兵也指不出列表路径。**这是它的真实天花板,R2 须评估。** |
| **候选② 独立 daemon 持 key 自动翻页,agent 只读落盘** | **refactor(条件接受 · 不是 cut)** | 它**能**真正解决接口截断(daemon 持 key 直接打列表 API 翻页)。代价=引入**全新信任边界 daemon↔agent**。关键设计:daemon 必须**完全在 agent context 之外**(独立进程、独立凭据存储、agent 既不读 daemon 的 key 也不能让 daemon 执行 agent 给的任意请求),agent 只读 daemon **已落盘**的结果——本质是把「被动性即隔离」从「operator 手动浏览」换成「daemon 自动翻页」,隔离性质**可以保留**,但**信任主体从人变成了一个持 key 的程序**,这是质变,须协议层兜。 |
| **候选③ agent 出翻页计划 + operator 一键确认 + 持 key 进程执行** | **new(强候选 · 与①并列)** | 最优雅:agent 只产「参数序列」(纯数据,无凭据),operator 审一眼确认,持 key 进程(可以是 daemon,也可以是 operator 触发的一次性脚本)执行。**agent 不持 key、不直接发请求**的保证最干净(agent 的输出是 declarative plan,不是 imperative request)。代价=operator 每批要确认一次,自动化程度低于②但高于①。 |
| **§2件1 措辞**(扩展到采集会话凭据隔离) | **refactor(倾向 yes)** | 见 §1B。把隐式/外推的约束显式成文,给自动翻页一个成文依据。改法是**新增子条款**而非改既有 scope。 |
| **C5 scope 配套动** | **keep(初判无需动)** | C5 管的是「采什么内容 + 内容往哪流」,自动翻页是「怎么采到列表」——**采集手段**变化,**采集对象**(operator 有权访问的顾问内容)没变,**自用不传播**没变。初判 C5 **不需要**因自动翻页改。但 R2 须确认「程序化翻页」会不会被 operator 的合规判断认为触及 C5「不规避访问控制」的边界(那是 operator 的事,但措辞要不要显式区分「被动 vs 主动」留个口)。 |
| **速率/源白名单治理** | **new(但可能属 L4)** | 自动翻页一旦主动发请求就有风控/速率问题。是否本轮定还是 L4 定,R2 决。 |

**初步立场倾向(strong-converge 要求最终单一,这里先给方向不 hedge)**:我倾向 **③为主 + ①兜底**的混合——
③把「持 key 发请求」与「agent 编排」用 declarative plan 干净切开,信任边界最清晰;①作为「列表 API 够得到
的内容」的低成本默认。**②(常驻 daemon)我持保留**:常驻持 key 进程的攻击面 + 生命周期管理 + 「daemon 会
不会被诱导执行 agent 构造的请求」是比③更重的协议负担,除非接口截断严重到非常驻翻页不可,否则③的「一次性
持 key 进程按确认过的 plan 执行」已够。**§2 需新增采集会话凭据隔离子条款**(把隐式约束显式化)。

## §3 · 我现在最不确定的 3 件事

1. **列表接口截断 2025-07-21 的根因未知 → 直接决定①是否可行**。backlog 给的是现象(列表止于某 contentId,
   2026 仅 detail 可达),没给根因:是**分页 cursor 还没翻到**(那②/③翻页能解)、还是**列表 API 服务端就
   不返回 2026 项**(那翻页也没用,2026 只能靠 detail 入口逐篇)?**这是整个 verdict 的事实地基** —— 若是
   后者,①侦察兵和②③翻页**都够不到列表**,真正要做的是「detail 入口的批量发现」而非「列表翻页」。X 标的
   无法回答此根因,须 R2 由 operator/实战数据澄清。
2. **「持 key 主体」与 agent 的物理隔离怎么做到可验证**。③说「agent 出 plan、持 key 进程执行」,但 plan
   里若含 URL/参数,持 key 进程拿 plan 去发请求 —— 怎么保证 agent 不能通过构造恶意 plan 让持 key 进程发
   越权请求?「agent 不持 key」容易,「agent 不能滥用持 key 进程」难。这需要持 key 进程对 plan 做**白名单
   校验**(只接受预定义的列表/detail 端点 + 受限参数范围),而不是盲执行 agent 给的任意 URL。这条边界
   X 标的没有现成机制,是协议修订草案必须补的核心。
3. **§2 改动 vs 仅 backlog 记录的边界**。K 第 3 点问「是否触及 §1/§2 协议层措辞修订」。我初判倾向 yes
   （需补采集会话凭据隔离子条款),但不确定 operator 是想要**协议级硬约束**(写进 SHARED-CONTRACT §2,
   对所有 XenoDev 采集类项目生效),还是**项目级约束**(只写 008-pB PRD/spec)。前者改动面大、收益是「以后
   所有采集项目受益」,后者轻、但下次别的采集项目撞同样问题。这关系到协议修订草案的**层级定位**,R2 须定。
