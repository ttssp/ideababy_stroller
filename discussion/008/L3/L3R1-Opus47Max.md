## 0. How I read the intake

008 是 004 的**上游数据采集模块**:把一个指定微信投资顾问每天发的内容(图文分析、盘中预警、直播回放)
自动、不漏、省心地收进 operator 的个人知识库,供 004 决策消费。operator 接受 2-3 个月 / 高投入,
宁可慢宁可复杂也要"不漏 + 省心"。这把 scope 的取舍方向定死了:**遇到"漏一点但简单" vs "不漏但复杂",
永远选不漏;遇到"要人盯" vs "全自动但难做",选全自动**。

- **Hard constraints I'm respecting**:008=004 的采集模块(输出对 004 友好/边界清晰/不做通用器);
  2-3 月 15-30h/周;永久自用;红线 = 不产建议 + 不二次分发;优先级 = 不漏 + 省心。
- **Unknowns I'll propose options for**:① 采集落点(本机/云/手机微信)② 内容形态范围 ③ 知识库深度
  ④ 给 004 的输出契约。
- **Red lines I'll honor**:不产生投资建议/买卖信号;不二次分发/公开顾问内容。(intake 已明确,§5 我再补 2 条候选软红线供确认。)

**三个候选的差异轴**:**v0.1 覆盖多少内容形态 × 知识库沉淀做多深**。A 窄而深(只图文,但沉淀到结构化可喂 004);
B 全形态但浅(图文+预警+回放都收,先存原始);C 折中(图文+预警,跳过最重的回放)。三者都是真 peer ——
不是"做不做",而是"先吃哪块、沉淀到哪层"。技术怎么采集**不在本轮**(那是 L4)。

## 1. Candidate A · "图文优先 · 沉淀到能喂 004"

### v0.1 in one paragraph
只做一件事但做透:自动采集顾问的**图文分析**(盘前/盘后,每天最稳定、最高频的内容),不只是存下来,
而是**结构化沉淀**(日期、标的、顾问观点要点、原文),做到 004 能直接读。盘中预警和直播回放 **v0.1 不做**。

### User persona (sharpened from L2)
operator 本人:已付费订阅该顾问,每天产出盘前盘后图文分析。他不想再手动翻看/复制,想要一个"每天自动
把这些分析整理进库、按标的可查、随时能喂给 004"的底座。他能接受 v0.1 暂时收不到盘中预警(自己临时盯一下)。

### Core user stories (3-5)
- 作为 operator,我能让程序每天自动收下顾问的盘前盘后图文分析,**不用我手动操作**。
- 作为 operator,我能按**标的/日期**检索"顾问对 XX 股说过什么"。
- 作为 operator,我能把某段时间的顾问观点**直接导出/喂给 004** 做决策参考。
- 作为 operator,当采集失败(源头变化/登录失效)时,我能**收到通知**而不是默默漏掉。

### Scope IN
- 单一顾问、单一内容形态(图文分析)的自动采集
- 结构化沉淀:日期 + 标的 + 观点要点 + 原文留存
- 按标的/日期检索
- 对 004 友好的导出格式(粗约定即可)
- 采集失败告警(呼应"不漏")

### Scope OUT (explicit non-goals)
- ❌ 盘中预警(时效强、触发不规律 → 留 v0.2)
- ❌ 直播回放(形态最重:视频/音频转写 → 留 v1)
- ❌ 多顾问/多平台(永远不在 v0.1)
- ❌ 任何投资建议生成(红线 → 是 004 的事)

### Success looks like (observable outcomes)
- 连续 2 周,顾问发的图文分析**零漏采**(漏了有告警)。
- operator 能在 <30 秒内查到"上周顾问对某标的的观点"。
- 导出的数据 004 能直接读入,无需 operator 手工转换。

### Honest time estimate under human's constraint
- 15-30h/周 × 2-3 月:这个窄切其实**用不满**,大概 3-5 周能到可用。**但**真正的时间黑洞是
  "自动且不失效"(登录态维持/源头变化适配)—— 这部分可能吃掉剩余时间,且**置信度低**。
- Confidence: **M**。不确定点全在采集落点(见 §4 ❓1)—— 落点没定,工期估计都是浮的。

### UX principles (tradeoff stances, not designs)
- **不漏 > 全面**:宁可只收图文,也要把图文收到一条不漏 + 失败必告警。
- **省心 > 功能**:operator 介入次数趋近于零是第一目标;检索/导出做够用即可。
- **结构化 > 原样堆**:既然定位是喂 004,沉淀就值得做到结构化(这是 A 区别于 B 的核心)。

### Biggest risk to this cut
A 把"沉淀到结构化"放进 v0.1,赌的是"图文分析能被可靠地结构化提取"。如果顾问图文格式很随意
(截图为主、无规整文本),结构化成本可能远超预期,反而拖累"快速拿到一个不漏的采集器"这个更基础的目标。

## 2. Candidate B · "全形态收录 · 先存原始不求结构化"

### v0.1 in one paragraph
v0.1 就追求一件事:**一个都不漏**。顾问发的**所有**形态——图文、盘中预警、直播回放——v0.1 全收,
但**只做原始留存 + 时间线归档**(回放存视频/音频文件,不转写;图文存原样;预警按时间记)。先保证"全量、
不漏地进库",结构化/检索/喂 004 留到以后。

### User persona (sharpened from L2)
operator 本人,最怕"漏了关键内容"——尤其盘中预警和某次直播里的重要判断。他宁愿先有一个"啥都给我收下来、
按时间堆好"的全量归档,哪怕暂时只能自己翻,也不接受任何一种形态被漏掉。

### Core user stories (3-5)
- 作为 operator,顾问发的**任何**内容(图文/预警/回放)都被自动收下,**零形态遗漏**。
- 作为 operator,我能按**时间线**回溯某天顾问发过的所有东西。
- 作为 operator,采集任一形态失败时我立刻知道。
- 作为 operator,我能本地打开任意一条归档的原始内容(看图/读文/播回放)。

### Scope IN
- 单一顾问、**全内容形态**自动采集(图文 + 盘中预警 + 直播回放)
- 原始留存 + 时间线归档(回放=存文件不转写)
- 全形态采集失败告警
- 本地浏览原始归档

### Scope OUT (explicit non-goals)
- ❌ 结构化提取(标的/观点要点)→ 留 v0.2
- ❌ 按标的检索 / 全文搜索 → 留 v0.2
- ❌ 喂 004 的格式契约 → 留 v0.2(v0.1 先确保数据进来了)
- ❌ 直播回放转写(文字化)→ 留 v1
- ❌ 投资建议生成(红线)

### Success looks like (observable outcomes)
- 连续 2 周,**三种形态全部零漏**(任一漏了有告警)。
- operator 能按日期翻出"那天顾问发的全部内容",包括能播放回放。
- operator 对"我不会再漏掉任何顾问内容"有信心(主观但关键)。

### Honest time estimate under human's constraint
- 15-30h/周 × 2-3 月:**刚好吃满甚至偏紧**。难点不在量,而在**三种形态三套采集逻辑** + 尤其直播回放
  (大文件、可能有时效链接)。
- Confidence: **M-L**。回放采集的可行性强依赖落点(手机/微信环境?见 §4 ❓1),置信度被它拖低。

### UX principles (tradeoff stances, not designs)
- **不漏 > 一切**:B 把"不漏"推到极致——覆盖所有形态,哪怕每种都只做最浅的留存。
- **先进库 > 先好用**:v0.1 接受"只能堆着翻",把结构化/检索的复杂度整体推后。
- **原样保真 > 加工**:先把原始证据完整留下,加工是后话(也避免提取出错丢信息)。

### Biggest risk to this cut
B 赌"三种形态都能在 v0.1 内搞定采集"。直播回放是最大变数——如果它在微信小程序里、链接有时效、或必须
手机端操作,光这一项就可能撑爆 2-3 个月。**B 的"全"可能让 v0.1 迟迟无法收尾**,违背"先有个能用的不漏采集器"。

## 3. Candidate C · "图文 + 预警 · 跳过最重的回放"

### v0.1 in one paragraph
折中:收**图文分析 + 盘中预警**这两种**文本/图文类、时效性最相关**的内容,**跳过直播回放**(形态最重、
最不确定)。沉淀做到**轻结构化**(时间 + 标的标签 + 原文),够检索、够粗喂 004,但不追求 A 那样的深度提取。

### User persona (sharpened from L2)
operator 本人。他最在意的其实是"**别漏掉有时效的判断**"——盘前盘后图文 + 盘中预警就覆盖了绝大多数
对决策有用的、且有时效的内容。直播回放更多是"复盘补充",晚几个月再收也不致命。

### Core user stories (3-5)
- 作为 operator,顾问的图文分析**和盘中预警**都被自动、及时收下,不漏。
- 作为 operator,盘中预警到达时我能**及时知道**(它最有时效价值)。
- 作为 operator,我能按标的/日期查图文和预警。
- 作为 operator,采集失败有告警。
- 作为 operator,我能把这些喂给 004(轻量格式)。

### Scope IN
- 单一顾问、两种形态(图文分析 + 盘中预警)自动采集
- 轻结构化沉淀(时间 + 标的标签 + 原文)
- 按标的/日期检索
- 盘中预警的及时通知(呼应其时效性)
- 采集失败告警 + 轻量 004 导出

### Scope OUT (explicit non-goals)
- ❌ 直播回放(整体留 v1 —— 形态最重)
- ❌ A 那样的深度结构化(观点要点抽取)→ 轻结构化够用,深提取留 v0.2
- ❌ 多顾问/多平台
- ❌ 投资建议生成(红线)

### Success looks like (observable outcomes)
- 连续 2 周,图文 + 预警**零漏**;预警到达有及时通知。
- operator <30 秒查到某标的的图文/预警历史。
- 轻量导出能进 004。

### Honest time estimate under human's constraint
- 15-30h/周 × 2-3 月:**舒适落在区间内**,留出余量应对"自动不失效"这个真难点。
- Confidence: **M+**。比 A/B 高——砍掉回放就砍掉了最大的不确定性;剩下的图文+预警同源同形态,复杂度可控。

### UX principles (tradeoff stances, not designs)
- **覆盖有时效的 > 覆盖全部**:优先保住"漏了就没价值"的内容(预警),放弃"晚收无妨"的回放。
- **省心 > 深加工**:轻结构化够检索够喂 004,不为深提取增加维护面。
- **及时 > 实时**:预警要"及时通知"但不需毫秒级。

### Biggest risk to this cut
C 假设"图文和预警同源、采集逻辑可复用"。如果盘中预警的发布渠道/形态跟图文差异很大(比如预警走推送、
图文走小程序页面),那 C 的"两形态"可能并不比 B 省多少。需在 L4 验证两者是否真的同源。

## 4. Options for the human's ❓ items

### ❓1 · 采集落点 / 运行环境(⭐ 头号未决,直接决定可行性)
intake 平台三选,我读为"落点未定"。三条路 product 含义:
- **Option 1 · 本机常驻(Mac)**:最简单、最可控。代价:电脑关机/睡眠就不采 → **与"不漏"直接冲突**。
  适合 operator 电脑基本常开的情况。
- **Option 2 · 云/服务器常驻**:7x24 不漏。代价:多一层运维成本(违背"省心"一点),且**能否在云端
  访问微信小程序内容是大问号**(可能根本拿不到源)。
- **Option 3 · 手机/微信环境采集**:因顾问内容在微信小程序,源头**可能必须**是手机/微信侧。最贴近真实
  数据源,但通常最不"省心"(手机要常在、可能要特定 app 环境)。
> **我的判断**:这不是"选一个"那么简单,而是**"不漏 + 省心"这对最高优先级在这里直接打架**——
> 最不漏的(云 7x24)可能拿不到源;最能拿到源的(手机)最不省心。**这是 008 全局成败的关键决策,
> 必须在进 L4 前 human 拍板**,且可能需要先做一个"源头到底怎么访问"的技术探针(L4 第一件事)。
> L3 scope 无法替它做决定,但应把它标成"动手前必答 + 高风险"。

### ❓2 · 内容形态范围(= 选 A/B/C 的实质)
- Option 1(=A):只图文 → 最快有"不漏的采集器",但暂时漏预警/回放。
- Option 2(=B):全形态 → 最不漏,但 v0.1 可能收不了尾。
- Option 3(=C):图文+预警,跳回放 → 覆盖有时效内容,工期可控。**我个人倾向 C**(理由见 §3 置信度)。

### ❓3 · 知识库深度
- Option 1 · 原始留存(B 的做法):先堆着,结构化后置。最快进库,但暂不好用、暂难喂 004。
- Option 2 · 轻结构化(C 的做法):时间+标的标签+原文。够检索够粗喂 004。
- Option 3 · 深结构化(A 的做法):抽观点要点。最好喂 004,但提取成本高、依赖图文规整度。

### ❓4 · 给 004 的输出契约
- Option 1 · v0.1 就粗约定一个格式(参 004 已 ship 的输入)→ 008 一交付就能接 004。
- Option 2 · v0.1 先不定,只保证数据完整留存,契约留 v0.2 → 降 v0.1 复杂度,但 008 暂时喂不了 004。
> 鉴于定位是"004 的采集模块",我倾向 **Option 1 至少粗约定**,否则"模块"定位落空。

## 5. Red lines I'd propose(intake 已明确 2 条,这里补 2 条软红线供确认)

intake 已定:不产投资建议 / 不二次分发。补充建议确认:
1. **v0.1 单源(只一个指定顾问)** — 因桥接 doc §4 + "004 模块"定位:做成通用采集器会立刻失控。建议升为硬边界。
2. **不动顾问账号的"写"权限** — 008 只读取/采集 operator 自己能看到的内容,绝不代发/代操作账号。
   (合规 + 安全双重考虑;intake 未提但值得钉死。)

## 6. Questions that need real user interviews (L3 can't answer)

008 自用,"用户访谈"= operator 自我确认 + 少量源头实测:
- **源头实测**:顾问内容在微信小程序里**到底怎么访问**?(决定 ❓1,可能需先做技术探针——这是唯一真正
  挡路的未知,且偏 L4)
- 顾问图文分析的**实际格式**有多规整?(决定 ❓3 深结构化是否现实 → A 是否可行)
- 盘中预警和图文是否**同源同渠道**?(决定 C 是否真的比 B 省 → C 的核心假设)
- 直播回放的链接**有无时效**、能否离线留存?(决定回放该不该进 v0.1 → B 的最大风险)
