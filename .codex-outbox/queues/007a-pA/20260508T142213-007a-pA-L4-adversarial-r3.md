## R3 修订引入的新问题

- **C14 still contradicts D17/D20.** The hard constraint row at `spec.md:80` still says review session must use off or an equivalent review-mode alias. That conflicts with `spec.md:104-105`, `T012.md:37-38`, and `T031.md:73-80`.
- **`friction --on` does not clear review_mode.** `T012.md:48-51` only sets enabled for `--on`; a user who exits review by habit with `--on` can remain stuck with hook suppressed while O3 still passes. README teaches `--review-mode-off`, but the CLI should warn or clear explicitly.
- **Long-lived review_mode can hide schema drift.** `T011.md:55-60` exits before judge/schema validation when review_mode is true. That is correct for a short edit session, but if review_mode persists through reboot or day-14 launchd, O3 passes and schema_error_count may stay at 0 while capture is paused.

## New blockers (R3 引入或 R1+R2 漏抓的)

1. **Location:** `specs/007a-pA/spec.md:80`  
   **Issue:** Hard C14 still uses the old review-mode-as-off wording.  
   **Why it blocks:** C14 is a hard constraint and can instruct a build worker to implement the exact state collision R3 is meant to fix. This reopens B-R2-3 at the contract level.  
   **Fix direction:** Rewrite C14 to say review-mode sets `state.review_mode=true`, does not touch `enabled`, exits via `--review-mode-off`, and O3 ignores review_mode.

## New high-severity concerns

- `--on` should either clear `review_mode` or print a blocking warning when review_mode remains true. Otherwise the most natural recovery command can leave the hook paused.
- trust_summary should surface current `review_mode` status in Activity/Schema health or warning text, even if O3 ignores it. Otherwise review_mode stuck true is too easy to miss.

## Verdict

**BLOCK**

The R3 design direction is sound: independent `review_mode` fixes the O3 collision, and most propagation issues are closed. However, the hard C14 constraint still carries the old alias semantics, which is exactly the kind of stale contract wording that caused R2 to block. Because build workers treat §3 constraints as authoritative, this should go to R4 rather than ship-with-risk.

R4 should be a narrow final pass: update C14 and the remaining review-mode alias references, replace the last prescriptive `health_check` mention, and add a small `--on`/review_mode guard or explicit warning. I do not recommend shipping with the current C14 contradiction because R4 is cheap and this is the last hard cap.
