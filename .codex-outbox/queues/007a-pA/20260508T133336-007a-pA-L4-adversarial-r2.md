## New blockers (R2 引入或 R1 漏抓的)

1. **PRD v1.1 budget sign-off did not propagate to the build contract.** PRD C1 is now 12-23h with explicit 2026-05-08 sign-off, but spec frontmatter still says PRD v1.0, spec C1 still says 10-20h, spec §2 still claims 11.5-16.5h, and HANDOFF Step 3 still tells ADP C1 is 10-20h. This directly breaks the new R2 PRD change and will make ADP/spec consumers enforce the wrong hard constraint.

2. **O3 now has two incompatible definitions inside spec.md.** Outcomes and verification say current `enabled=true` is the only O3 gate. Scope IN-8 still defines the trust-summary hook-enabled state as enabled AND recent 24h agent activity. A worker following §2 can reintroduce the exact B3 bug while still citing the spec.

3. **Review-mode and O3 semantics are unresolved.** C14 requires `friction --review-mode`, and T012 implements it as the same state change as `--off`. That means a compliant review session can set `enabled=false`; if day-14 trust_summary runs during review, O3 fails even though the operator followed the review safety flow. The spec needs a separate review suppression state or an explicit O3 rule for temporary review-mode.

## New high-severity concerns

- **Read removal is not globally closed.** architecture §5.1 still lists Read in the hook contract, while §5.2/T010/T030 reject it. This is a high-risk ambiguity because architecture is the module contract most likely to guide T011.

- **HANDOFF completeness is undermined by stale values.** The mini-PRD block has the right sections, but C1 is stale. Cross-repo handoff is exactly where stale hard constraints become expensive.

- **T011 empty-stdin behavior is contradictory.** The pseudocode increments `schema_error_count` on `JSONDecodeError`, but the same task says empty stdin is a legal noop and must not increment. In Python, empty stdin normally produces `JSONDecodeError`; this needs an explicit pre-read empty check.
