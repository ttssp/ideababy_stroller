## 3. 用户视角真实状态(刺眼的事实)

**v0.1 ship 当日,用户跑 `./scripts/start.sh` 后**:

| 路径 | 状态 |
|---|---|
| `GET /healthz` | 200 ✅(@app.get 直接定义,不走 plugin registry) |
| `GET /docs` / `/redoc` / `/openapi.json` | 200 ✅(FastAPI 默认,paths 里只有 /healthz) |
| `GET /` 主页 | **404** ❌ |
| `GET /onboarding` | **404** ❌(模块没被 import,plugin registry 空) |
| `GET /decisions/new` | **404** ❌ |
| `POST /decisions/draft` | **404** ❌ |
| 其他所有业务路径 | **404** ❌ |

**根因**:`src/decision_ledger/main.py` 不导入 `decision_ledger.ui` 也不导入任何 `router_*` 模块。所有 `register_router(...)` / `register_startup_task(...)` 副作用注册依赖**模块被 import** 才会运行。production 路径下没人 import,plugin registry 永远是空的。`apply_to_app(app)` 在 main.py:166 被调,但此时 registry 内容为 0。

**为什么 v0.1 还能"完成"**:
- `tests/e2e/e2e_app_factory.py::make_e2e_app()` 手工 import 各 router 模块 + 调 `set_recorder()` 等 setter + `app.include_router(...)` — **bypass 整个 main.py lifespan**
- spec §6 O5 verification 用 e2e 测试通过即可,**没有承诺** `python -m decision_ledger.main` 启动路径在 v0.1 用户可用

**这个事实在哪可见**:
1. 启动 banner stderr(`_build_v01_banner` 第 5 条)— 用户每次 `./start.sh` 都会看到
2. `docs/known-issues-v0.1.md` §8(详细症状/根因/v0.2 计划)
3. `tests/integration/test_main_app_route_snapshot.py`(snapshot 测试,v0.2 接通时会自然 fail 提示更新)

**临时试用决策录入的方法**(不用 production):
