### DRIFT-1 · IDS L4 设计假设错误 — IDS 不应跑 SDD 工具链

**严重程度**: 架构级(不是字段级 drift,是 SHARED-CONTRACT v1.x + IDS CLAUDE.md 共同的根本错误)
**发现日期**: 2026-05-08(W0 收官 + W1 跨仓 hand-off 准备时,通过第一性原理推导得出)
**发现场景**: 第一性原理(`不受工作量约束的 应然分析`)

#### 现象

IDS CLAUDE.md L24 声明:"L4 · Plan — spec, architecture, tasks, parallel build, quality gates → output: working code + ship"。这句话隐含假设 IDS 仓自带完整 SDD 工具链(spec 验证 / task 调度 / pytest / Makefile / pre-commit)。

**实证不成立**:
- IDS 仓 **无 Makefile** (`No such file or directory`,2026-05-08 实证)
- IDS 仓 **无 pyproject.toml** (`No such file or directory`,2026-05-08 实证)
- IDS 仓 **无 pytest 配置 / no conftest.py / no testpaths**
- IDS 仓 **无 check-disjoint.sh / 无 spec_validator.py**
- IDS `.claude/commands/plan-start.md` 调 spec-writer + task-decomposer subagent 直接写 specs/ 目录,**只产文件,不真跑任何 build / test / 验证工具**

W0 期间 plan-start 跑通是因为它**只写文件,没真 build** —— 假设的错误一直没暴露。今天准备 W1 跨仓 hand-off 时撞墙:既然 IDS 跑不动 SDD 工具,L4 build 阶段必须迁到 ADP,但 SHARED-CONTRACT v1.1.0 §3 hand-off 协议假设 spec.md 已经在 IDS 这边产出,这两个事实**直接矛盾**。

#### 第一性结论

**应然分工**:
- IDS = idea → PRD(L1/L2/L3 + 可选 L4 hand-off 包),不写 specs/,不写 projects/
- ADP = PRD → spec → build → ship(完整 L4 SDD 工具链),自带 specs/ + projects/

**理由**(每条独立成立):
1. spec 形态(7 元素 schema / 9 字段 task / pre-commit 校验)与 ADP schema 演化绑定 → spec 物理在 ADP,schema 升 v0.2→v0.3 时同步改 spec 才原子
2. build 工具(Makefile / pytest / check-disjoint)在 ADP → spec 在 IDS = 跨仓跑工具 = 永远 drift
3. spec 内容来自 PRD,但 PRD 跟 spec 是不同 schema(IDS 8 字段 vs ADP 7 元素+frontmatter),两套 schema 各自正确,**hand-off 协议本来就要解耦** —— 这就是 SHARED-CONTRACT 存在的根本必要性
