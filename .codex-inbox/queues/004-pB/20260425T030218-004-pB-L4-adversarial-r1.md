## 核心挑战点 (必 chase)

按重要性降序 (前 3 条是**最硬 challenge**, 必出 finding):

### C1 · 录入 < 30 秒 UX 真的能达到吗? (对应 PRD Risk #1 最硬风险)

- D13 的"默认填 + 1/2/3/4 快捷键 + Enter 提交"组合是否足以保证 **wall-clock < 30s**?
- architecture.md §3.3 的"录入路径无 LLM 调用" — 真的能做到吗? Devil's advocate 占位 (每次 action 前 LLM 一句反驳) 是在录入路径内还是外? 如果在外, 用户怎么在 < 30s 内看到?
- T009 Playwright timing test 能否覆盖真实 keypress latency + 人类 decision time?
- search 证据: > 20 分钟一周弃用, > 30 秒在自用场景会退化为日志. **如果你能找出任何一条可能让单次录入超过 30 秒的路径, 这就是 BLOCKER**

### C2 · 三路冲突报告真的不偷偷给默认优先级吗? (红线 #10)

- StrategyModule IDL 的 `conflict_resolve(signals) → ConflictReport`: **谁实现它**? 如果 `agent_synthesis` (agent 综合) 承担 conflict_resolve 主体职责, 这是不是 **偷偷给了 agent 默认优先级**?
- 三个占位实现 (advisor / placeholder_model / agent_synthesis) 在 IDL 上**平权吗**?
- UI 的"三列表"排版顺序是否固定? 如果固定顺序且最后一列是 "agent 综合建议", 是否暗示 agent 是最终结论?
- **红线#11 "不动是一等输出"**: decision schema `action ∈ {buy/sell/hold/wait}` 中 `hold` 和 `wait` 是否真的在 UI 一等展示? 还是被归到"保持现状"这种 hidden default?

### C3 · 策略模块真的完全解耦吗? (红线 #9, 🧭 原则)

- 三个占位实现是否共享 private helper 导致耦合?
- `StrategyModule` ABC 的依赖范围是否"干净" (只依赖 PRD-level domain types, 不依赖 LLMClient / DB / Telegram)?
- 测试能否**单独测每个占位实现**, 不加载其他两个?
- v0.5+ 插入真私有 ML 模型时, 真的只需换一个 `CustomStrategyModule` 吗? 还是会碰到"placeholder 实际上和 conflict_resolve 绑死"的反模式?

### C4 · 失败告警 O10 + B-lite 降级路径完整性

- risks.md OP-1 mitigation 链路 (C11 < 30s + O5 timing test + O10 "连续 2 周 < 2 条告警" → B-lite) 是否真的形成闭环?
- T020 (失败告警监控) 触发后 human 的**下一步动作**是什么? 是 UI 告警, 还是 runbook, 还是自动降级?
- T022 B-lite runbook 是"只文档"还是有**可执行 toggle**? 如果只文档, human 在情绪低迷时可能不会真执行

### C5 · Upkeep 负担 — 6-8 周后停用最大失败模式

- PRD 明确: 没有外部 forcing function, 一旦 upkeep 断, 没人逼改进
- spec 有哪些**主动 counter-upkeep** 机制? 除了录入 < 30s, 还有什么让 human "想继续用" 的机制?
- 笔记 wiki (T016) 自动去重 — 真的不会在第 2 周因 agent 重复解释"右侧布局"而让 human 觉得"这 agent 没学会我的风格"?
- 周度 review (T014) 和月度 review (T015) 的**打开成本**是多少分钟? 是否会变成新的"作业感"?

### C6 · Telegram 通知真的节制吗? 不会变焦虑机吗? (红线 #4)

- C6 每周 1 次周报 + event 触发 — event 定义是什么? 如果 "event" 包含盘中 >3% 异动, 盘中每天都可能触发 5-10 次 → 变焦虑机
- T017 Telegram bot 的 event 过滤逻辑是否严格限定"仅 human 持仓相关 event + 仅市场时间内"?
- 有没有**muting 窗口**设计 (非市场时间不推)?

### C7 · 咨询师 pipeline (D9 半自动 watched folder) 现实可行性

- PDF/文本解析 (T006) 真的 < 2h/周吗? (R5 维护时间 ≤ 3 小时/周)
- 如果咨询师 PDF 格式突然变化 (5% 概率/月), 系统会静默失败还是显式报错?
- watched folder 假设 human 还在用 Proxyman 手工 fetch — 这个依赖的可靠性?

### C8 · PRD faithfulness 检查

- spec.md §1 每个 O1-O10 是否对应 PRD §6 O1-O10? 有没有被"软化"或"硬化"?
- spec 中的 Prior Decisions 是否真的来自 PRD / L2 / L3 而非 spec-writer 的猜测?
- 11 条 `[new in spec]` 的工程决策 (D8/D9/D10/...D20) 是否都合理, 有没有偷偷扩大 scope?
- scope OUT (non-goals.md 17 条) 是否真的覆盖 PRD §5 所有 OUT 项?

### C9 · Non-goal 泄漏检查

- spec 中有没有混进 PRD Scope OUT 的东西?
  - 事件日历 / FOMC 预警 (应 v0.2+, non-goal A)
  - 私有模型真训练 (应 v0.5+, 只接口占位)
  - 音频 / 视频解析 (应 v0.2+, 只占位)
  - 自动下单 (红线 R1, 永远不做)
  - 广谱 research terminal 功能 (A 的 scope, 非 B 的)

### C10 · Task DAG 合理性

- 22 个 task 有无循环依赖?
- file_domain 有无重叠 (除 depends_on 串行)?
- 并行宽度是否合理? T020 (失败告警监控) 真的能和 T019 / T021 并行吗 (都涉及 Phase 3 脚本)?
- 关键路径 74h + 串行 177h 与预算 100-160h: 超 17h (10%) 的预算风险 — spec 或 task 层面哪里可以砍?

### C11 · Solo-operator 实际可执行性

- 一个人 + AI 助手 5-6 周内真的能把 22 个 task 做完吗?
- 哪些 task 最可能 blocked on LLM prompt 调优 (T006 / T007 / T013 / T014 / T015)?
- 哪些 task 最可能 blocked on UI iteration (T008 / T010 / T011)?

### C12 · Solo-use 没有合规, 但有哪些 **personal 安全边界**?
