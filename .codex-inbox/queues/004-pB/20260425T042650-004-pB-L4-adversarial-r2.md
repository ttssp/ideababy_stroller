## 新审查点 (R2 引入的)

### N1 · cache miss 例外
spec-writer 在 R2 报告承认: cache 命中常态全程 < 30s, miss 时 35-40s. 主导场景 < 5% miss. spec.md Q11 标 OPEN.

**你的判断**: 这是 acceptable 边角情况, 还是破坏 PRD §S1 "≤ 30 秒表单" 承诺?

候选思考方向:
- PRD §S1 原文 "≤ 30 秒表单, 看到 agent 拉出的三路冲突报告" — 如果 cache miss 时 user 等 10s 看到报告, 这一段是否应该计入 30s? PRD 没写明.
- 缓解: cache 预热 (D11 + ConflictCacheWarmer) 是否真的能保证 < 5% miss? D9 流程末尾 enqueue 30-50 关注股的预生成是否真的覆盖典型录入场景?
- 退路: 若你认为是 BLOCK, R2 → R3 修向 — 把"报告生成 5-10s"显式列入"录入流程", 而非"≤30s 内"

### N2 · 三列顺序 hash deterministic per-day
spec-writer 提出候选改进: deterministic per draft_id 而非 per day.

**你的判断**: per-day hash 是否够 (理由: 跨天变化已经避免心理"最后一列 = 结论"); 还是必须 per draft_id?

### N3 · pause_pipeline façade graceful no-op
spec-writer 主动提出: T020 façade 对 T015/T017 是 graceful (import 失败 = no-op).

**你的判断**: 这是 robust 工程模式 (placeholder pattern), 还是隐藏 bug 风险 (静默失败可能让 pause 无效)?

### N4 · lane 隔离用 AST 静态分析
spec-writer 提出隐忧: dependency injection 绕过 AST 检测.

**你的判断**: 当前 IDL 设计是否真的不允许 lane 拿到其他 lane 实例 (registry 不 expose)? 若可以, 是否需补 runtime 检查 (e.g. 在 IDL ABCMeta 中拦截)?

### N5 · 总估时 181h vs 预算 160h
即使 cut list 也只到 168h. spec-writer 建议 human 至少砍 T019 LLM 抽取 (-3h) 落到 165h. **你判断**: 是否需在 R2 verdict 中要求 BLOCK 直到时间预算可信落入 100-160h, 还是 CONCERNS (可带风险 ship)?
